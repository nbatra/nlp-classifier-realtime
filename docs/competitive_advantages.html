Our IAB Audience Generation system delivers two structural advantages that compound over time and are difficult for competitors to replicate without re-architecting their platforms:

Pillar 1 — Compact Probability Vectors: By storing a single numerical probability vector per user instead of materialized segment memberships, we reduce KV-store storage by 3–8x, eliminate write amplification, and decouple audience segment creation from storage capacity. 💾 Part 1: Compact Probability Vectors — The Storage Revolution

1.1 How Traditional Adtech Audience Storage Works

To understand why our approach is fundamentally more efficient, we first need to understand the dominant storage model used by most audience platforms in programmatic advertising today.

Traditional platforms store audience data as explicit segment memberships. When a user qualifies for an audience segment (e.g., "Travel Enthusiast," "Finance Intender," "Auto Buyer In-Market"), a record is created that associates the user with that segment. Each record typically contains:

Traditional Segment Membership Record: { "user_id": "u-abc-123-def-456", // 36 bytes (UUID) "segment_id": "seg-travel-enthusiast-v3", // 30 bytes "segment_name": "IAB Travel Enthusiast", // 25 bytes "confidence": 0.82, // 8 bytes (double) "source": "behavioral", // 12 bytes "first_qualified":"2026-03-15T09:30:00Z", // 24 bytes "last_refreshed": "2026-04-08T14:00:00Z", // 24 bytes "expiry": "2026-05-08T14:00:00Z", // 24 bytes "taxonomy": "IAB-2.2", // 8 bytes "tier": "premium", // 8 bytes "ttl_days": 30 // 4 bytes } Approximate size per record: ~200–300 bytes (with key overhead, encoding, and index entries) Average segments per user: 15–40 segments (across behavioral, contextual, demographic, and lookalike segments) Total per user: 200 bytes × 25 segments (avg) = ~5,000 bytes (5 KB)

This model has been the industry standard because it maps cleanly to how advertisers think: "Give me all users in segment X." It is optimized for segment-level reads (which users are in this segment?) rather than user-level reads (what segments does this user belong to?). But this optimization creates significant storage and operational overhead.

The Scaling Problem with Materialized Segments

The fundamental issue is that storage scales as O(Users × Segments). Every new segment that a user qualifies for creates a new record. As the number of audience segments grows (to serve more advertiser verticals, finer targeting, custom audiences), storage per user grows linearly:

Scenario Segments per User Storage per User Storage for 500M Users Annual KV-Store Cost (Redis)
Basic (Tier-1 IAB only) 8–12 ~2.5 KB ~1.25 TB ~$50K/month
Standard (Tier-1 + Tier-2) 15–25 ~5 KB ~2.5 TB ~$100K/month
Advanced (IAB + custom + lookalike) 30–50 ~10 KB ~5 TB ~$200K/month
Enterprise (full taxonomy + partner segments) 50–100+ ~20 KB ~10 TB ~$400K/month
⚠ The Write Amplification Problem: Beyond raw storage, materialized segments create a write amplification problem. When a user visits expedia.com, a traditional platform must: (1) determine which segments the user now qualifies for (Travel Enthusiast, Hotel Intender, Adventure Traveler, Vacation Planner, etc.), (2) write or update each qualifying segment record independently, (3) check expiry/TTL on existing segments, and (4) potentially remove segments the user no longer qualifies for. A single page visit can trigger 5–15 write operations across the segment store. At 580K events/second, this means 3–9 million segment writes per second — a massive operational burden.

1.2 Our Approach: The Probability Vector Model

Instead of materializing segment memberships, we store a single compact probability vector per user. This vector contains one floating-point score and one timestamp per IAB category — approximately 700 entries total. The vector is the generating function from which any audience segment can be derived on demand.

Our Probability Vector Record (per user): { "user_id": "u-abc-123-def-456", // 36 bytes (UUID) "scores": { "IAB-1": { "s": 0.22, "t": 1712580000 }, // 8 bytes (float32 score + uint32 timestamp) "IAB-3": { "s": 0.15, "t": 1712566200 }, // 8 bytes "IAB-5": { "s": 0.93, "t": 1712696400 }, // 8 bytes "IAB-12": { "s": 0.85, "t": 1712580600 }, // 8 bytes "IAB-13": { "s": 0.55, "t": 1712580600 }, // 8 bytes "IAB-20": { "s": 1.85, "t": 1712739600 }, // 8 bytes "IAB-20-1":{ "s": 1.24, "t": 1712739600 }, // 8 bytes "IAB-20-3":{ "s": 1.56, "t": 1712739600 }, // 8 bytes ... // Only non-zero categories stored (sparse representation) } } Dense representation: 700 categories × 8 bytes = 5,600 bytes (5.6 KB) Sparse representation: Avg ~80–150 non-zero categories per active user = 640–1,200 bytes + 36 bytes key With compression (LZ4): ~400–800 bytes per user Typical active user: ~600 bytes compressed

Why Sparse Representation Works

Although the IAB taxonomy has ~700 categories, the average user does not visit websites spanning all of them. Empirical data from production bidstreams shows:

User Activity Level % of User Base Non-Zero IAB Categories Compressed Vector Size
Light (1–5 domains/day) ~45% 15–40 ~200–350 bytes
Moderate (5–20 domains/day) ~35% 40–120 ~350–700 bytes
Heavy (20–50+ domains/day) ~15% 120–250 ~700–1,200 bytes
Power (50+ domains/day) ~5% 250–400 ~1,200–2,000 bytes
Weighted Average ~80–150 ~600 bytes
Key Insight — Exponential Decay Helps Sparsity: Our exponential decay mechanism naturally prunes stale scores toward zero. Categories the user has not engaged with recently decay below the storage threshold and are removed from the sparse vector during periodic cleanup. This means the vector stays compact over time — it does not grow unboundedly even for long-lived user profiles. Decay is not just an accuracy feature; it is also a storage efficiency feature.

1.4 The Write Amplification Deep Dive

Write amplification is arguably a bigger operational advantage than raw storage savings. The following traces a single user event through both systems to see the difference.

Scenario: User visits expedia.com

The domain lookup returns scores for: IAB-20 (Travel: 0.91), IAB-20-1 (Adventure Travel: 0.72), IAB-20-3 (Hotels: 0.68), IAB-3 (Business: 0.15).

❌ Traditional System — Write Path

  1. Evaluate segment rules for "Travel Enthusiast" → user qualifies → WRITE segment record
  2. Evaluate rules for "Hotel Intender" → qualifies → WRITE
  3. Evaluate rules for "Adventure Traveler" → qualifies → WRITE
  4. Evaluate rules for "Vacation Planner" → qualifies → WRITE
  5. Evaluate rules for "Travel & Leisure" (broad) → qualifies → WRITE
  6. Evaluate rules for "Business Traveler" → below threshold → skip
  7. Evaluate rules for "Luxury Travel" → below threshold → skip
  8. Check if user should be removed from any segments that conflict or have expired → potential DELETE operations

Total: 5 writes + rule evaluation overhead per event

At 580K events/sec: ~2.9M write operations/sec

✅ Our System — Write Path

  1. Look up domain scores from in-memory table (0.5–1.5ms)
  2. Read user's probability vector from Redis (1–2ms)
  3. Add domain scores to vector: scores[IAB-20] += 0.91, scores[IAB-20-1] += 0.72, etc.
  4. Update timestamps for modified categories
  5. Single WRITE of updated vector back to Redis

Total: 1 write per event, always

At 580K events/sec: 580K write operations/sec

Write Amplification Impact at Scale: Traditional system: 580,000 events/sec × 5 writes/event = 2,900,000 writes/sec Daily write volume: 2.9M × 86,400 = 250.6 Billion writes/day Our system: 580,000 events/sec × 1 write/event = 580,000 writes/sec Daily write volume: 580K × 86,400 = 50.1 Billion writes/day Reduction: 5x fewer write operations Impact on infrastructure: • Redis cluster can be sized for 580K writes/sec instead of 2.9M writes/sec • Fewer nodes needed (6–8 primary shards vs. 25–30) • Lower network bandwidth between application and Redis • Reduced replication traffic (1 write replicated vs. 5) • Lower risk of write-related latency spikes during traffic peaks

1.5 Zero-Cost Segment Creation

Perhaps the most strategically important advantage of the probability vector model is that creating new audience segments has zero marginal storage cost. This changes the economics of audience monetization entirely.

Traditional Platform: Adding a New Segment

1
Define segment rules: "Premium Travel Intender" = users with Travel score > X who also have Finance score > Y (high-value travelers). Rules engine must be updated.
2
Backfill existing users: Scan all 500M user profiles to determine who currently qualifies. Write new segment records for qualifying users. This takes 4–12 hours on a batch compute cluster (Spark/EMR).
3
Storage allocation: If 50M users qualify, that is 50M × 250 bytes = ~12.5 GB of new segment records. Redis/Aerospike capacity must be provisioned.
4
Ongoing maintenance: The real-time pipeline must now evaluate this new segment's rules for every incoming event. Write amplification increases by 1 additional write for every qualifying event.

Our Platform: Adding the Same Segment

1
Define segment derivation rule: "Premium Travel Intender" = users where decayed_scores[IAB-20] > 1.5 AND decayed_scores[IAB-13] > 0.5. Add rule to the segment derivation config (a JSON file, ~100 bytes).
2
Deploy: Push config update. No backfill, no batch job, no storage change. The rule is evaluated at read time against the existing probability vector. Qualifying users appear in the segment immediately.
Time to activate a new segment: Traditional platform: 4–12 hours (backfill + deploy). Our platform: minutes (config push + rolling restart). This speed advantage is commercially significant — when an advertiser asks "can you create a custom segment for our campaign launching tomorrow?", the answer is always "yes, give us 10 minutes."

⏱ Part 2: Real-Time Intent Detection — Winning Bids Earlier and Cheaper

2.1 The Information Latency Gap Explained

In programmatic advertising, the value of an audience signal is not just about accuracy — it is equally about timing. A perfectly accurate audience signal that arrives 24 hours late is worth dramatically less than a moderately accurate signal that arrives in milliseconds. This is because of how real-time bidding (RTB) auctions work.

How RTB Auctions Determine Clearing Prices

In a second-price auction (the dominant RTB model), the winning bidder pays slightly above the second-highest bid. The clearing price is therefore determined by how many bidders recognize the user's value:

Scenario Bidders with Intent Signal Competition Level Typical Clearing Price
Only our DSP has the signal 1 out of 8–12 bidders Low — other bidders see untargeted impression, bid base CPM $2.50–$4.00 CPM
2–3 platforms have the signal 2–3 out of 8–12 bidders Moderate — some competition from informed bidders $5.00–$7.00 CPM
All major platforms have the signal 6–8 out of 8–12 bidders High — most bidders competing with audience multipliers $8.00–$14.00 CPM
Key Insight — Auction Dynamics Reward Early Detection: The clearing price is set by the second-highest bidder, not the highest. When only one bidder (us) knows the user is a travel intender, the second-highest bid comes from an uninformed bidder paying base CPM ($2–3). When all bidders know, the second-highest bid comes from another informed bidder paying audience-adjusted CPM ($8–12). The same impression, for the same user, costs 3–4x more once the signal becomes widely available. The window of exclusivity is our profit margin.

2.4 The Compounding Effect: Early Signals Improve Campaign Optimization

The advantage of real-time detection extends beyond individual impression costs. It creates a compounding feedback loop in campaign performance optimization.

1
Earlier data → earlier optimization: DSP bid algorithms learn from conversion feedback. When our DSP wins impressions earlier in a user's intent journey, it receives conversion signals (clicks, site visits, purchases) hours sooner. This feedback accelerates the campaign optimization loop — the DSP learns which user+context combinations convert and adjusts bids faster.
2
Lower CPMs → more impressions per budget: An advertiser with a $50K daily budget buying at $3.80 CPM (our early-window price) gets ~13.2M impressions. The same budget at $9.50 CPM (post-window price) gets ~5.3M impressions. More impressions means more data points for optimization, more chances to reach the right users, and better frequency distribution.
3
Higher win rate → better frequency capping: When competition is lower, our DSP wins a higher percentage of auctions it enters. This means better control over ad frequency — we can show an ad exactly 3 times to a user (optimal for brand recall) rather than showing it once (insufficient) or 15 times (wasteful) because we keep losing auctions and compensating by over-bidding.
4
Better campaign performance → more advertiser spend: Advertisers measure return on ad spend (ROAS). Campaigns that achieve better ROAS attract larger budgets from the same advertiser and attract new advertisers through case studies and word-of-mouth. This creates a revenue flywheel: better data → better performance → more spend → more bid data → even better signals.

🧠 Part 3: Combined Strategic Analysis

3.1 Why These Advantages Are Structurally Linked

The storage efficiency and real-time detection advantages are not independent — they are structurally coupled. You cannot have real-time scoring at 580K QPS without a compact, single-write storage model. And the compact storage model naturally requires real-time score updates (because you are storing raw probabilities, not pre-computed segments). The architecture produces both advantages simultaneously.

The Causal Chain

Compact probability vectors (600 bytes/user) └─► Fits in small Redis cluster (6 nodes vs. 54) └─► Single write per event (no write amplification) └─► Can sustain 580K writes/sec on modest infrastructure └─► Real-time score updates become feasible └─► Intent detected in milliseconds, not hours └─► DSP bids with exclusive signal └─► Lower CPMs during early window └─► Better ROAS for advertisers └─► More advertiser spend └─► More revenue to reinvest The compact storage model ENABLES real-time scoring. Real-time scoring REQUIRES compact storage. They are two sides of the same architectural coin.

3.3 Risk Factors and Mitigations

⚠ Risk 1: Competitor Adopts Similar Architecture
A well-funded competitor (e.g., The Trade Desk, LiveRamp) could build a similar real-time scoring system from scratch. Mitigation: Our head start means we are accumulating user-level intent data and advertiser relationships while they are building infrastructure. The flywheel compounds. Also, our system already works — theirs does not yet exist. First-mover advantage in data products is significant.
⚠ Risk 2: Privacy Regulations Limit User-Level Scoring
Stricter privacy regulations could restrict user-level probability vectors. Mitigation: Our system already supports cohort-level scoring (see Privacy section in system overview). The probability vector model works equally well at the cohort level — instead of per-user vectors, we maintain per-cohort vectors. The storage and real-time advantages still apply.
⚠ Risk 3: Auction Dynamics Shift to First-Price
If all exchanges move to pure first-price auctions (trend is ongoing), the early-detection advantage changes shape. In first-price auctions, we still benefit from knowing user intent (we can bid appropriately), but the mechanism for capturing value shifts from lower clearing prices to more efficient bid shading. Mitigation: Real-time signals are equally valuable for bid shading algorithms — knowing intent earlier means our bid shading model has better inputs, leading to more efficient spend.

  • Speed: Intent detected in milliseconds vs. hours. 12–24 hour exclusivity window against batch competitors. 47% lower effective CPMs during the early window.
  • Agility: New audience segments activated in minutes with zero storage cost. Enables same-day custom segments for advertiser RFPs that batch platforms simply cannot match.

These advantages are structurally linked — you cannot have real-time scoring without compact storage, and compact storage naturally enables real-time updates. Competitors cannot adopt one advantage without the other, and neither can be retrofitted onto a traditional segment-store architecture without a ground-up rewrite. This makes the moat durable and deepening over time.
