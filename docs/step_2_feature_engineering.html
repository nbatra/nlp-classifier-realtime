🧪 2.2 Text Vectorization — Why TF-IDF Over Alternatives

The most critical feature engineering decision in this system is how we convert crawled website text into numerical feature vectors for domain classification. A librarian assigning every book in a massive library to one of 700 genre categories could read every book cover-to-cover and develop a deep semantic understanding (like BERT), or quickly scan for distinctive keywords that reliably signal genre — "recipe," "preheat," "tablespoon" almost certainly means Cooking. TF-IDF takes the keyword-scanning approach, and for this use case, it is dramatically more effective per compute dollar.

Decision: TF-IDF Vectorization

📋 Technology Decision: Text Feature Extraction Method

Problem: Convert crawled website text (avg 5,000 words per page) into fixed-size numerical vectors for a classifier that maps domains to ~700 IAB content categories.

Chosen: TF-IDF with n-grams (unigrams + bigrams), vocabulary size 50,000

✅ Why TF-IDF Wins

  • Inference speed: Sparse matrix multiply — ~10ms per document on CPU
  • Interpretable: Top features are human-readable terms like "mortgage rate" or "flight booking"
  • Minimal infrastructure: No GPU cluster needed; scikit-learn on standard VMs
  • Batch-friendly: Vectorize 200K domains in <2 hours on 8 cores
  • Sparse storage: ~500 non-zero values out of 50K dims = 99% sparse = ~2 KB per domain
  • Strong baseline accuracy: TF-IDF + Linear SVM achieves 85-90% top-1 accuracy on IAB classification (industry benchmarks)
  • Debuggable: When classification is wrong, inspect top TF-IDF terms to understand why

❌ Limitations (Acceptable)

  • No semantic understanding — "car" and "automobile" are separate features
  • Bag-of-words loses word order
  • Cannot handle unseen vocabulary (OOV words get 0 weight)
  • Bigrams partially mitigate but don't fully solve context issues
  • Vocabulary must be periodically refreshed as web content evolves

Rejected Alternative 1: Word2Vec / FastText Embeddings

❌ Rejected: Word2Vec / FastText Averaged Document Embeddings

Approach: Train or use pre-trained word embeddings (300-dim), average all word vectors in a document to get a document-level representation.

✅ Potential Benefits

  • Captures semantic similarity ("car" ≈ "automobile")
  • Dense 300-dim vector — smaller per-domain storage
  • Handles some OOV words (FastText subword approach)

❌ Why We Rejected It

  • Information loss: Averaging word vectors collapses a 5,000-word document into 300 dims — loses topic-discriminative keywords
  • Poor for multi-topic pages: A CNN.com homepage about politics AND sports averages to a meaningless middle ground
  • Not interpretable: Cannot inspect "why was this domain classified as Travel?"
  • Marginal accuracy gain: Benchmarks show <2% improvement over TF-IDF for topic classification tasks, not worth the complexity
  • Requires pre-trained model management: Must version, store, and load embedding matrices (~1.2 GB for 4M-word FastText)

Rejected Alternative 2: BERT / Transformer Embeddings

❌ Rejected: BERT-based Document Embeddings

Approach: Use a pre-trained BERT model (or fine-tuned variant) to encode crawled page text into 768-dim contextualized embeddings.

✅ Potential Benefits

  • State-of-the-art semantic understanding
  • Captures context, negation, nuance
  • Transfer learning from massive pre-training corpus

❌ Why We Rejected It

  • Inference cost: ~200ms per document on GPU, ~2 seconds on CPU — vectorizing 200K domains would take 111 GPU-hours (~$350 per batch run)
  • 512 token limit: BERT truncates at 512 tokens; our crawled pages average 5,000 words. Sliding window approaches add 10x complexity
  • GPU infrastructure required: Need persistent GPU cluster for weekly re-classification — adds $15-30K/month in infrastructure
  • Overkill for topic classification: BERT excels at nuanced NLU tasks (sentiment, entailment). For "is this page about Travel or Sports?" keyword signals suffice
  • Not interpretable: 768-dim dense vector provides no insight into classification reasoning
  • Diminishing returns: Studies on topic classification show BERT improves over TF-IDF+SVM by ~3-5%, but at 50-100x the compute cost

Rejected Alternative 3: LLM-based Classification (GPT-4, Claude)

❌ Rejected: LLM API Classification

Approach: Send crawled page text to a large language model API with a prompt like "Classify this webpage into IAB categories" and parse the response.

✅ Potential Benefits

  • Highest accuracy for ambiguous/edge cases
  • Zero training data needed — works out-of-the-box
  • Can provide reasoning for each classification

❌ Why We Rejected It

  • Cost: ~$0.01-0.03 per domain classification × 200K domains = $2,000-6,000 per weekly batch — vs ~$5 for TF-IDF+SVM on CPU
  • Latency: 2-10 seconds per classification — 200K domains = 55-278 hours even with parallelization
  • Rate limits: API rate limits restrict throughput to ~1,000-3,000 requests/minute
  • Non-deterministic: Same input can produce different outputs across API calls, making evaluation and debugging unreliable
  • Vendor lock-in: Entire classification pipeline depends on third-party API availability and pricing
  • Privacy concerns: Sending crawled website content to external APIs may violate data processing agreements
Note on Hybrid Approaches: We do use LLMs selectively for labeling the training set (see Step 4: Model Training). A small set of ~5,000 domains is labeled using LLM classification to bootstrap the training data. But the production classification pipeline uses TF-IDF + traditional ML classifier for the reasons above. The LLM is a labeling tool, not a production inference tool.

Stage 1: Text Extraction and Cleaning

Raw crawled HTML is messy. A typical webpage contains navigation menus, ad scripts, cookie banners, footers, and boilerplate that has nothing to do with the page's topical content. If we vectorize all of this text, our TF-IDF features will be dominated by noise terms like "cookie policy," "subscribe," "all rights reserved," and "privacy."

1

HTML Tag Stripping and Boilerplate Removal

We use a content-extraction library (trafilatura or readability-lxml) rather than naive HTML tag stripping. These libraries use heuristic rules to identify the "main content" block and discard navigation, sidebars, headers, and footers.

Language Detection and Filtering

We classify extracted text using langdetect (or fastText language ID). For v1, we process English-only content. Non-English domains are assigned IAB categories using URL/metadata heuristics only (covered in Section 2.4).

Text Normalization

After extraction, we apply a normalization pipeline to reduce vocabulary size and improve feature quality:

Normalization steps (in order):
  1. Lowercase:           "New York Flights" → "new york flights"
  2. Remove URLs:         "visit http://..." → "visit"
  3. Remove emails:       "contact@site.com" → ""
  4. Remove numbers:      "save $299 on flights" → "save on flights"
     (exception: keep numbers in specific patterns like "5-star", "top-10")
  5. Remove punctuation:  "world-class" → "world class"

  6. Remove short tokens: tokens < 2 characters dropped
  7. Collapse whitespace: multiple spaces → single space

Result: Clean, lowercase, alphabetic text ready for tokenization

Stage 2: Tokenization and N-gram Generation

4

Tokenization Strategy

We use whitespace + punctuation tokenization (not subword/BPE) because TF-IDF operates on complete meaningful terms. We do NOT apply stemming or lemmatization — here is why:

Why no stemming/lemmatization:
  - "flights" and "flight" → "flight" (stemmed) — OK, these are synonymous
  - "cooking" and "cook" → "cook" (stemmed) — OK for topic signal
  BUT:
  - "university" and "universe" → "univers" (Porter stemmer) — WRONG,
    conflates Education and Science
  - "organization" and "organ" → "organ" (aggressive stemmer) — WRONG,
    conflates different IAB categories

For IAB topic classification, we WANT the vocabulary to distinguish
"automotive" from "automation" and "finance" from "fine dining."
Stemming reduces vocabulary by ~30% but introduces ~5% category confusion.

Decision: No stemming. Accept larger vocabulary; rely on IDF weighting
to handle related forms. Bigrams capture multi-word topic signals.
5

N-gram Generation: Unigrams + Bigrams

We generate both unigrams (single words) and bigrams (two-word phrases). Trigrams were tested but rejected.

Example: "book cheap flights to new york from los angeles"

Unigrams: ["book", "cheap", "flights", "new", "york", "from", "los", "angeles"]
Bigrams:  ["book cheap", "cheap flights", "flights new", "new york",
           "york from", "from los", "los angeles"]

Why bigrams matter for IAB classification:
  "new york"      → Travel/Destinations (unigram "new" alone is useless)
  "real estate"   → Real Estate (unigram "real" alone is ambiguous)
  "credit card"   → Personal Finance
  "climate change" → Science/Environment
  "machine learning" → Technology/Computing
  "video games"   → Gaming

Why NOT trigrams:
  - Vocabulary explodes: 50K unigrams + bigrams → 250K+ with trigrams
  - Most trigrams are too specific / low-frequency to be useful
  - Tested: trigrams added ~0.5% accuracy but increased vocab 5x
  - Decision: Not worth the storage and computation cost

Stopword Removal

We use a custom stopword list that extends the standard English stopwords with web-specific terms:

Standard English stopwords (NLTK, ~180 words):
  "the", "is", "at", "which", "on", "a", "an", "and", "or", ...

Web-specific additions (~120 words):
  "click", "subscribe", "share", "comment", "login", "signup",
  "cookie", "privacy", "terms", "copyright", "reserved", "menu",
  "search", "home", "contact", "about", "next", "previous", "page",
  "advertisement", "sponsored", "partner", "affiliate", ...

Domain-specific exclusions (words we KEEP despite being common):
  "free"        → Keep: signals Free Software, Freemium services
  "best"        → Keep: signals review/comparison content
  "top"         → Keep: signals ranking/list content
  "news"        → Keep: critical for News/Current Events IAB category
  "health"      → Keep: critical for Health & Fitness category

Total stopword list: ~300 words
Impact: Reduces token count per page by ~40%, vocabulary by ~8%

Stage 3: Vocabulary Selection

After tokenization, we have millions of unique terms across 200K domains. We must select a fixed vocabulary of 50,000 terms that maximizes classification signal while keeping computation tractable.

Vocabulary Selection Criteria

Raw vocabulary after tokenization of 144K English domains:
  Total unique unigrams:    ~2.1 million
  Total unique bigrams:     ~18.4 million
  Combined:                 ~20.5 million unique terms

Selection process:
  1. Minimum document frequency (min_df=5):
     Remove terms appearing in fewer than 5 domains
     → Eliminates typos, domain-specific jargon, proper nouns
     → Reduces to ~850K terms

  2. Maximum document frequency (max_df=0.85):
     Remove terms appearing in >85% of domains
     → Eliminates near-universal terms missed by stopword list
     → Reduces to ~780K terms

  3. Select top 50,000 by TF-IDF score sum across corpus:
     For each surviving term, sum its TF-IDF weight across all documents
     → Ranks terms by overall discriminative power
     → Selects final 50,000 vocabulary

Why 50,000 dimensions:
  - 30,000: Tested — loses ~2% accuracy due to missing niche IAB terms
  - 50,000: Sweet spot — captures topic-discriminative terms across all 700 IAB cats
  - 100,000: Tested — gains <0.3% accuracy, doubles storage and compute
  - 200,000: Dominated by low-frequency terms that add noise

Final vocabulary composition:
  Unigrams:   ~35,000 (70%)
  Bigrams:    ~15,000 (30%)

Stage 4: TF-IDF Computation

With the vocabulary fixed, the next step is computing TF-IDF weights. The math here matters — these formulas determine the quality of every downstream feature.

TF-IDF Formula

TF-IDF(t, d) = TF(t, d) × IDF(t)

Where:
  TF(t, d)  = Term Frequency of term t in document d
  IDF(t)    = Inverse Document Frequency of term t across corpus

TF computation (sublinear):
  TF(t, d) = 1 + log(raw_count(t, d))    if raw_count > 0
           = 0                             if raw_count = 0

  Why sublinear? A term appearing 100 times is not 100x more important
  than appearing once. The log dampens frequency inflation from verbose pages.

IDF computation (smoothed):
  IDF(t) = log((1 + N) / (1 + df(t))) + 1

  Where:
    N     = total number of documents in corpus (144,000 English domains)
    df(t) = number of documents containing term t

  The +1 smoothing prevents division by zero and ensures terms appearing
  in all documents still get non-zero (though low) IDF weight.

L2 normalization (per document):
  v_normalized = v / ||v||₂

  This ensures that long documents and short documents produce vectors
  of equal magnitude — preventing a 10,000-word page from dominating
  a 500-word page purely due to length.

Sparse Vector Storage

The 50,000-dimensional TF-IDF vector for each domain is extremely sparse — on average, only ~500 of the 50,000 positions are non-zero. Storing it as a dense float32 array would waste 198 KB per domain. Instead, we use sparse representation:

Sparse Storage Format

Dense representation:  50,000 × 4 bytes = 200,000 bytes (200 KB) per domain
Sparse representation: ~500 non-zero × (4 bytes index + 4 bytes value) = ~4,000 bytes (~4 KB)
                       But with compressed sparse row (CSR): ~2 KB avg

Storage savings: 200 KB → 2 KB = 99% reduction

For 200K domains:
  Dense:  200K × 200 KB = 40 GB   ← Does not fit in a single Redis node
  Sparse: 200K × 2 KB   = 400 MB  ← Easily fits in a single Redis node

Sparse format used (Protocol Buffers):
  message DomainTfIdfVector {
    string domain = 1;
    repeated uint32 indices = 2 [packed=true];  // vocabulary indices
    repeated float values = 3 [packed=true];    // TF-IDF weights
    int64 computed_at = 4;                      // Unix timestamp
    string vocab_version = 5;                   // Vocabulary hash
  }

Cyclical Time Encoding: Why Sin/Cos Over One-Hot

Time features (hour-of-day, day-of-week) are inherently cyclical — 11pm is close to midnight, Sunday is close to Monday. One-hot encoding treats these as unrelated categories, losing the cyclic proximity. We use sin/cos encoding instead:

Cyclical Time Encoding

Hour of day (0-23):
  hour_sin = sin(2π × hour / 24)
  hour_cos = cos(2π × hour / 24)

Day of week (0-6, Monday=0):
  dow_sin = sin(2π × day / 7)
  dow_cos = cos(2π × day / 7)

Example calculations:
  Midnight (hour=0):   sin=0.000,  cos=1.000
  6am (hour=6):        sin=1.000,  cos=0.000
  Noon (hour=12):      sin=0.000,  cos=-1.000
  6pm (hour=18):       sin=-1.000, cos=0.000
  11pm (hour=23):      sin=-0.259, cos=0.966  ← Close to midnight!

Why this matters for IAB:
  - 11pm and midnight users have similar browsing patterns (entertainment,
    social media). One-hot would treat them as maximally distant.
  - Friday evening and Saturday morning shoppers are similar audiences.
    Cyclical encoding preserves this proximity.

Dimensionality comparison:
  One-hot hour: 24 dimensions
  Cyclical hour: 2 dimensions  ← 12x reduction, better distance metric

SSP Category Signal Integration

OpenRTB bid requests often contain SSP-declared content categories in the site.cat and site.pagecat fields. These are IAB category codes declared by the supply-side platform (SSP) or publisher. We integrate these as supplementary signals:

1

Extract SSP categories from bid request

Parse site.cat (site-level) and site.pagecat (page-level) arrays. Map IAB category codes to our internal taxonomy IDs.

2

Compute confidence-weighted blend

SSP categories are often inaccurate (publishers self-declare, sometimes for ad revenue optimization). We weight them below our own classification:

final_iab_score[category] =
    0.80 × our_classification[category]    // TF-IDF classifier output
  + 0.15 × ssp_cat_signal[category]        // SSP-declared (binary → 1.0 if declared)
  + 0.05 × bcat_inverse_signal[category]   // Inverse bcat (if blocked, domain is likely adjacent)

Why 80/15/5 weighting:
  - Our classification: Trained on crawled text, covers all 700 categories, most reliable
  - SSP signal: Often accurate but sometimes gaming/miscategorized (~70% agreement rate)
  - bcat signal: Weak but free signal; advertisers blocking "Gambling" on a site
    suggests the site has Gambling-adjacent content
3

Handle unknown domains via SSP-only classification

For domains not in our pre-computed lookup table (~15% of bid request domains), SSP categories become the primary signal for the user score update, weighted at 1.0 instead of 0.15.

⚠ Challenge: SSP Category Gaming

Some publishers deliberately miscategorize their sites (e.g., declaring a low-quality content farm as "News") to attract higher-CPM ad campaigns. Our defense:

  • Cross-SSP validation: If the same domain is declared as different categories by different SSPs, flag as unreliable
  • Our classification as ground truth: When our TF-IDF classifier disagrees with the SSP category, we trust our classifier (80/15 weighting)
  • Drift monitoring: Track agreement rate between our classification and SSP declarations over time. Alert if agreement drops below 60%

👤 2.6 User-Level Features — IAB Score Arrays and Behavioral Signals

User-level features are the ultimate output of the feature engineering pipeline. Every domain classification and every bid request enrichment ultimately flows into a per-user IAB score array that represents the user's interest profile across ~700 IAB categories. This score array is what the audience segmentation system reads to decide which segments a user belongs to.

Think of it as a fingerprint of interests: a 700-dimensional vector where each dimension represents the user's affinity for one IAB category, accumulated from their browsing history and decayed over time.

Score Accumulation: How User Scores Update in Real-Time

When a bid request arrives for a known user visiting a classified domain, the user's IAB score array is updated. The update rule must be fast (computed in <3ms as part of the <10ms user score SLA) and must balance recency against historical signal.

Score Update Formula

On each bid request event for user U visiting domain D at time T:

  1. Look up domain IAB probabilities: P[D] = float32[700]
     (Pre-computed by TF-IDF classifier, stored in Redis)

  2. Apply confidence weighting:
     confidence = min(1.0, P[D][max_category])
     // High confidence when dominant category has probability > 0.5
     // Low confidence when categories are evenly split (ambiguous domain)

  3. For each IAB category i where P[D][i] > threshold (0.01):

     // Apply time decay to existing score first
     elapsed = T - iab_last_update[U][i]
     λ_i = decay_rate[i]    // Category-specific decay rate
     decayed_score = iab_scores[U][i] × exp(-λ_i × elapsed)

     // Add new signal
     iab_scores[U][i] = decayed_score + (confidence × P[D][i])

     // Update metadata
     iab_last_update[U][i] = T
     iab_event_count[U][i] += 1

Threshold of 0.01:
  - Without threshold: update all 700 categories for every event
    → 700 writes per event × 580K QPS = 406M writes/sec (too expensive)
  - With threshold: update ~15-30 categories per event (where P[D][i] > 0.01)
    → ~20 writes per event × 580K QPS = ~12M writes/sec (manageable)

Cross-Device Identity Resolution Features

A single person often uses multiple devices (phone, tablet, laptop, work computer). Without identity resolution, we would maintain 4 separate, sparse user profiles instead of 1 comprehensive one. The feature engineering implications are significant:

1

Identity Graph Lookup

For each bid request, resolve the device ID to a household/person cluster using a probabilistic identity graph (from partners like LiveRamp or built in-house from IP + user-agent co-occurrence patterns).

Input:  device_id = "GAID_abc123" (Android phone)
Graph lookup → cluster_id = "HH_98765"
Other devices in cluster:
  - "IDFA_def456" (iPhone — spouse?)
  - "cookie_ghi789" (Chrome on laptop)

Decision: Merge or keep separate?
→ We MERGE scores at the cluster level for audience segmentation
→ We KEEP individual device profiles for frequency capping
2

Score Merging Strategy

When merging scores across devices in a cluster:

For each IAB category i:
  cluster_score[i] = max(device_1_score[i], device_2_score[i], ...)

Why MAX (not SUM or AVERAGE):
  - SUM: Inflates scores for multi-device users, creating targeting bias
  - AVERAGE: Dilutes strong single-device signals
  - MAX: Preserves the strongest signal regardless of device
         If user researched Travel heavily on phone, that signal
         should be the cluster-level Travel score — not averaged
         with zero-Travel signals from other devices
⚠ Challenge: Cookie Deprecation and ID Loss

As third-party cookies are deprecated (Chrome, Safari already blocks), a growing fraction of users are unidentifiable across sessions. Feature engineering implications:

  • ~40% of web bid requests now arrive without a persistent user ID
  • For these users, we cannot accumulate score arrays over time
  • Fallback: Use contextual features only (domain IAB category + bid request features) for single-session audience assignment
  • Cohort-based approach: Group cookieless users by (domain, geo, device_type, hour) and assign the average IAB score array for that cohort
  • This degrades audience precision from per-user to per-cohort, but still provides value vs. no targeting

Technology Decision: Redis vs. Alternatives for Hot Tier

📋 Technology Decision: Hot-Tier Feature Serving

Problem: Serve domain IAB probabilities and user score arrays at <2ms latency for 580K+ QPS.

Chosen: Redis Cluster (separate clusters for domain and user data)

✅ Why Redis

  • Sub-millisecond reads for key-value lookups
  • Native support for binary data (HSET for score arrays)
  • Built-in TTL and LRU eviction for cache management
  • Cluster mode scales horizontally with automatic sharding
  • Atomic read-modify-write via Lua scripts (for score updates)
  • Battle-tested at AdTech scale by industry peers

❌ Rejected: Memcached

  • No built-in persistence — full cache rebuild on restart
  • No atomic read-modify-write for score updates
  • No cluster mode — requires client-side sharding
  • 1 MB value limit restricts future feature vector growth

❌ Rejected: DynamoDB

  • Median latency ~5ms (single-digit ms), too close to our SLA
  • Cost at 580K WCU + 580K RCU: ~$120K/month (vs $30K for Redis)
  • No in-memory guarantee — may hit cold reads

❌ Rejected: Aerospike

  • Excellent performance (sub-ms on SSD), but operational complexity higher than Redis
  • Smaller talent pool — harder to hire/onboard engineers
  • License costs add up at scale
  • Would consider for future if Redis memory costs become prohibitive

Technology Decision: ScyllaDB vs. Alternatives for Warm Tier

📋 Technology Decision: Warm-Tier User Profile Storage

Problem: Store 500M user profiles (~4.85 TB) with single-digit millisecond reads for cache misses and efficient batch scans for retraining.

Chosen: ScyllaDB (Cassandra-compatible, C++ implementation)

✅ Why ScyllaDB

  • 10x lower tail latencies vs Apache Cassandra (C++ vs JVM, no GC pauses)
  • Wide-column store perfect for our sparse score arrays (only store non-zero categories)
  • Linear horizontal scaling — add nodes as user base grows
  • TTL per column for automatic expired score cleanup
  • Excellent write throughput (580K+ writes/sec across 24 nodes)
  • CQL compatibility — familiar query language for engineers

❌ Rejected: Apache Cassandra

  • JVM garbage collection causes p99 latency spikes (50-200ms) under heavy write load
  • Requires 3-4x more nodes for same throughput as ScyllaDB
  • Compaction storms degrade read latency unpredictably

🔍 2.9 Feature Quality — Drift Detection, Staleness, and Degradation

Features degrade over time. Websites change content, user behavior shifts seasonally, and the web vocabulary evolves. Feature quality monitoring is not optional — it is the difference between an audience platform that maintains accuracy and one that silently degrades.

Feature Staleness Detection

Feature Expected Refresh Cycle Staleness Threshold Action if Stale
Domain TF-IDF vectors Weekly >14 days since last crawl Priority re-crawl queue; use existing classification with "low confidence" flag
Domain IAB probabilities Weekly >14 days since last classification Serve stale classification but reduce confidence weight by 50%
User IAB scores Real-time per event >30 days since last event Profile eligible for archival; exclude from active audience segments
IDF values (vocabulary) Monthly >60 days since last IDF recomputation Trigger vocabulary refresh pipeline; may indicate web content drift

Feature Drift Detection

Feature drift occurs when the statistical distribution of feature values changes over time, causing model accuracy to degrade even though the model itself has not changed. In our system, drift manifests in several ways:

1

Vocabulary Drift

New terms enter the web lexicon (e.g., "metaverse" in 2021, "GPT" in 2023) that are not in our TF-IDF vocabulary. These terms get zero weight, causing the classifier to miss emerging content categories.

Detection:
  - Track out-of-vocabulary (OOV) rate per batch run
  - Baseline OOV: ~8% of tokens in crawled text
  - Alert threshold: OOV > 12% → indicates vocabulary needs refresh
  - Metric: oov_rate = count(tokens not in vocab) / count(all tokens)

Mitigation:
  - Monthly vocabulary refresh: Recompute vocabulary from latest crawl data
  - Rebuild IDF values from updated corpus
  - Re-vectorize all domains with new vocabulary
  - Retrain classifier on new feature space (see Step 4)
2

Domain Content Drift

Websites change their content focus over time. A domain that was primarily Sports may pivot to cover Sports + Gambling (a common pattern with sports betting legalization).

Detection:
  - Compare this week's classification vs last week's for each domain
  - Flag domains where top-1 category changed
  - Flag domains where top-1 probability dropped by > 0.15

  Typical week:
    Domains with no category change:      ~96%
    Domains with minor shift (same top-1): ~3%
    Domains with major shift (new top-1):  ~1% (~2,000 domains)

  Action for major shifts:
    - Manual review queue for top 100 highest-traffic shifted domains
    - Automatic update for remaining 1,900 (trust new classification)
    - Alert if >3% of domains shift in a single week (unusual, investigate)
3

User Behavior Drift

Aggregate user browsing patterns change seasonally (more Travel in summer, more Shopping in Q4) and with world events (pandemic shifted away from Travel toward Home & Garden).

Detection:
  - Track weekly aggregate IAB category distribution across all user profiles
  - Compute Jensen-Shannon divergence (JSD) between this week and 4-week rolling average
  - Alert threshold: JSD > 0.05 → investigate

  Example seasonal pattern:
    Q4 (Oct-Dec): Shopping category share increases from 8% to 15%
    Summer (Jun-Aug): Travel category share increases from 5% to 12%
    These are expected — only alert on UNEXPECTED shifts

Mitigation:
  - Decay rates handle short-term shifts naturally (old signals fade)
  - For structural shifts, adjust category-specific decay rates
  - For category distribution rebalancing, see Step 6 (Monitoring)

Graceful Degradation Strategy

⚠ What Happens When Features Fail?

Feature serving infrastructure can fail in multiple ways. Each failure mode has a defined graceful degradation path:

Failure Mode Impact Detection Degradation Strategy Recovery Time
Redis domain cluster down No domain IAB lookups Health check fails, latency spikes Fall back to ScyllaDB (higher latency, 2-5ms). If ScyllaDB also down, use SSP categories only. Auto-failover: 30s. Full recovery: 5-15min
Redis user cluster down No user score reads/writes Health check fails Fall back to ScyllaDB for reads. Buffer score writes in Kafka, replay on recovery. Auto-failover: 30s. Write replay: 5-30min
Batch pipeline failure Stale domain classifications Pipeline monitoring, no fresh data in Redis Serve previous week's classifications (still valid, ~99% domains don't change weekly). Alert ops team. Manual re-run: 2-3 hours
Crawl infrastructure down No fresh crawl data for new domains Crawl job fails, S3 data missing Existing domain features remain valid. New/changed domains use SSP categories until crawl resumes. Crawl restart: 6-12 hours
ScyllaDB degraded performance Slow cache-miss path p99 latency > 20ms Increase Redis cache TTL to reduce cache misses. For new users, initialize with empty profile (no ScyllaDB read). Depends on root cause: minutes to hours
Key Insight — Defense in Depth: Every feature has multiple fallback data sources. Domain classification: our classifier → SSP categories → "Unknown." User scores: Redis → ScyllaDB → empty profile. This layered approach means the system never fully stops — it degrades in quality rather than availability. A user with an empty profile still gets contextual targeting from the domain's IAB category; it's just less personalized.
