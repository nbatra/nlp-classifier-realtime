Step 1: Data Engineering & Feature Pipeline

IAB Category Audience Generation for Programmatic Advertising

🎯 Core Objective

Build the data foundation that ingests 50 billion+ daily bid requests from programmatic ad exchanges, crawls 200,000+ curated domains for content classification, and maintains 500 million+ user score profiles — all feeding a real-time IAB audience generation engine that classifies users into ~700 IAB content categories for targeted advertising.

  • Bidstream Ingestion: Consume OpenRTB bid requests at 580K+ QPS with sub-second end-to-end latency
  • Domain Crawl Pipeline: Systematically crawl and extract text from curated high-traffic domains for offline classification
  • Data Quality Gate: Filter bot traffic, detect domain spoofing, validate consent signals, and ensure clean data reaches downstream models
  • Storage Architecture: Multi-tier storage optimized for real-time lookups (<5ms), streaming user score updates, and batch classification workloads
  • Pipeline Monitoring: End-to-end observability across ingestion, crawl, quality, and storage layers with automated alerting

Why This Matters: Every downstream component — TF-IDF vectorization, domain classification, user score accumulation, audience segment generation — depends on the quality and timeliness of data flowing through this pipeline. A domain misclassification due to stale crawl data or a missed bid request due to ingestion lag directly erodes the CPM premiums our audience segments command (well-classified segments earn 2-5x higher CPMs).

📊 1.1 Scale Reality Check

Before designing any pipeline, we need to confront the raw numbers. Programmatic advertising operates at a scale that trivially breaks systems designed for "millions" of records. Our platform participates in the real-time bidding (RTB) ecosystem, receiving bid requests from dozens of SSPs (Supply-Side Platforms) simultaneously.

The Layman's Analogy

A post office receiving 580,000 letters per second, where each letter contains information about a website a person just visited. The job: (a) open each letter, (b) check if it's legitimate or forged, (c) look up what category the website belongs to, (d) update a profile card for that person, and (e) file everything — all before the next 580,000 letters arrive one second later. That is the bidstream pipeline.

💡 Key Insight: The 84% Reduction

Of the 55 TB of raw bid request data ingested daily, only about 3.0 TB (roughly 5.5%) survives our quality gates as filtered events usable for score updates. The rest is discarded due to: bot traffic (~35%), missing/invalid user IDs (~25%), unconsented users under GDPR/CCPA (~12%), duplicate requests (~8%), and domains not yet in our classification table (~4%). This aggressive filtering is intentional — garbage in, garbage out is amplified at this scale, and every false score update degrades audience segment quality.

Why These Numbers Matter for Architecture Decisions

The scale numbers directly drive our technology choices:

  • 580K QPS sustained eliminates single-node message brokers and requires horizontally-scalable streaming (Kafka with partitioning)
  • 55 TB/day raw ingestion rules out writing raw bid requests to a transactional database — we need append-only log storage with tiered compaction
  • <5ms domain lookup requirement mandates in-memory key-value stores (Redis/Aerospike), not disk-based databases
  • 500M user profiles with concurrent read/write at 95K updates/sec pushes us toward wide-column stores (Cassandra/ScyllaDB) or sharded Redis clusters
  • 200K domains × bi-weekly crawl is a modest batch workload that fits on a small Scrapy cluster — no need for distributed crawl infrastructure

Our data engineering pipeline consumes four distinct data sources, each with different characteristics, refresh cadences, and quality concerns. Understanding the shape of each source is critical before we design ingestion.

🔔 OpenRTB Category Fields: Free Signal, Noisy Signal

About 40% of bid requests include publisher-declared IAB categories (site.cat). These are free in that they require no crawling or classification. However, they are notoriously unreliable: publishers often self-declare broad Tier-1 categories (e.g., "IAB1" for Arts & Entertainment) or leave them stale for years. Our strategy: use publisher-declared categories as a supplementary signal when our own domain classification is unavailable, but never as the primary source. When both are available, our crawl-based classification takes precedence, and publisher-declared categories serve as a confidence booster if they agree.

Source 2: Domain Crawl Data

Conceptual Approach: While bidstream data tells us which domains users visit, it does not tell us what those domains are about. To classify domains into IAB categories, we need the actual text content. We maintain a curated list of ~200K domains (covering ~96% of bidstream traffic by volume) and periodically crawl them to extract readable text.

What We Crawl vs. What We Skip

✅ Crawled (Curated List)

  • Top 200K domains by bid request volume
  • Domains with 100+ daily bid requests
  • New domains appearing in bidstream for 3+ consecutive days
  • Domains flagged for reclassification by drift detection

❌ Not Crawled

  • Long-tail domains with <100 daily bid requests (~4% of traffic)
  • Domains behind login walls (paywall, authentication)
  • Domains blocking our crawler via robots.txt
  • Known ad-fraud / MFA (Made-For-Advertising) domains

For uncrawled domains, we fall back to: (1) OpenRTB publisher-declared categories, (2) URL/domain-name heuristics, or (3) a "General" catch-all category with low confidence.

Source 4: OpenRTB Metadata (bcat / cat Fields)

Beyond the primary site/content categories, the OpenRTB protocol carries additional category metadata that we can mine:

  • bcat (Blocked Categories): Categories the advertiser does not want to appear alongside. If an advertiser blocks "IAB7" (Health & Fitness), this tells us something about campaign brand safety, not about the user — but aggregated across many bid requests, it reveals which categories are commercially sensitive (and therefore high-CPM).
  • site.cat / app.cat: Publisher-declared categories, as discussed above.
  • site.content.cat: Content-level categories, more granular than site-level. Available in ~15% of requests but more accurate when present.

💡 Source Priority Matrix

When multiple category signals are available for a domain, we use this priority order:

  1. Our crawl-based classification (highest confidence, freshest data)
  2. site.content.cat (content-level, more specific)
  3. site.cat (site-level, publisher-declared)
  4. URL heuristics (domain name keywords)
  5. "Unclassified" fallback (no score update for this event)

⚡ 1.3 Bidstream Ingestion Architecture

Ingesting 50 billion bid requests per day — sustaining 580K QPS with bursts to 1.2M QPS — is the most demanding engineering challenge in this pipeline. Every design choice in this section is driven by three constraints: throughput (never drop bid requests), latency (end-to-end processing under 2 seconds), and cost (storage and compute must be economically viable at 55 TB/day).

The Layman's Analogy

Think of our ingestion layer as a massive postal sorting facility. Trucks arrive from 30+ delivery routes (SSPs) simultaneously. Each truck drops off hundreds of thousands of letters per second. The facility must: sort letters into categories, stamp each with a received-at timestamp, throw away junk mail (bots), verify addresses (user IDs), and forward good letters to the right department — all without letting a single truck idle in the parking lot because the building is full.

Technology Decision: Message Broker

⚖️ Why Apache Kafka?

Problem: We need a message broker that sustains 580K+ messages/sec, supports multiple consumer groups reading the same data, retains messages for replay, and handles 1.2M QPS bursts without data loss.

✅ Apache Kafka (Chosen)

  • Proven at >1M msg/sec per cluster at LinkedIn, Uber, Netflix
  • Partitioned log model: 512 partitions give us fine-grained parallelism
  • Consumer groups allow independent readers (quality filter, archive, analytics)
  • Replayable: if a consumer fails, it can re-read from an offset
  • Tiered storage support: spill to S3 for retention beyond 24h
  • Mature ecosystem: Kafka Connect, Schema Registry (Avro), exactly-once semantics

❌ Rejected Alternatives

  • Amazon Kinesis: 1MB/sec/shard limit means we'd need 700+ shards; resharding is slow and disruptive; higher per-shard cost at our scale
  • Apache Pulsar: Excellent on paper (tiered storage native, multi-tenancy), but smaller operational community; our team has deep Kafka expertise
  • RabbitMQ: Not designed for this throughput; single-queue model lacks Kafka's partitioned parallelism; no built-in replay
  • Google Pub/Sub: Managed simplicity is appealing, but at 50B msgs/day the cost is prohibitive (~$150K/month for messaging alone); also, message ordering guarantees are weaker

Ingestion Fleet Design

The ingestion fleet is a horizontally-scaled pool of stateless HTTP servers behind an L4 (TCP) load balancer. Each server does minimal processing: parse, validate, extract, serialize, produce to Kafka. No business logic, no database calls, no blocking I/O beyond the Kafka produce.

Why Stateless?

  • Horizontal scaling: Add pods to handle traffic spikes; remove them when traffic subsides. Kubernetes HPA scales from 200 to 400 pods during Black Friday.
  • Fault tolerance: If a pod dies, the load balancer routes traffic to surviving pods within 5 seconds. No state to recover.
  • Deployment simplicity: Rolling deploys replace pods one at a time with zero downtime. No coordination needed.

🕑 Worked Example: A Single Bid Request's Journey

A user visits cnn.com/travel/best-beaches-2026 on their iPhone in Los Angeles. The SSP (Google Ad Manager) sends a bid request:

  1. T+0.0ms: Bid request arrives at NLB, routed to ingestion pod #147
  2. T+0.3ms: Pod parses the OpenRTB JSON (1.2 KB payload)
  3. T+0.4ms: Extracts fields: domain=cnn.com, ifa=6D92078A..., geo=US/CA, site.cat=[IAB20]
  4. T+0.5ms: Schema validation passes; ingestion timestamp assigned: 2026-04-04T14:23:01.847Z
  5. T+0.7ms: Serialized to Avro using Schema Registry (schema ID 42), compressed with Snappy
  6. T+1.2ms: Produced to Kafka topic bid-requests-raw, partition 317 (hash of domain + user_id)
  7. T+1.3ms: HTTP 204 returned to SSP — request complete

Total ingestion latency: 1.3ms. The bid request is now durably stored in Kafka, available for all downstream consumers.

Serialization: Why Avro over JSON?

Serialization Format Decision

✅ Apache Avro (Chosen)

  • Schema Registry enforces contract between producers and consumers
  • Binary format: ~60% smaller than JSON for our bid request payloads (1.2 KB vs 3 KB)
  • Schema evolution: add fields without breaking existing consumers
  • At 580K msg/sec, the 60% size reduction saves ~420 MB/sec of network bandwidth

❌ Rejected Alternatives

  • JSON: Human-readable but 2.5x larger; parsing overhead at 580K QPS adds ~200ms/sec of CPU per pod
  • Protocol Buffers: Slightly better compression than Avro, but schema evolution is less flexible (field renumbering issues); Avro's self-describing schemas work better for our analytics consumers
  • MessagePack: No schema enforcement; schema drift has caused production incidents in past projects

Backpressure and Overflow Strategy

Even with generous provisioning, traffic can exceed capacity during compound events (Black Friday + major sporting event + new SSP onboarding simultaneously). Our backpressure strategy:

1 Kafka Producer Buffering: Each ingestion pod buffers up to 64 MB of messages locally before blocking. This absorbs micro-bursts of 2-3 seconds duration.

2 Async Produce with Retry: Kafka producer uses acks=1 (leader acknowledgment) for speed. Retries up to 3 times with exponential backoff on transient failures.

3 Load Shedding: If Kafka produce latency exceeds 500ms (indicating cluster saturation), ingestion pods begin sampling: accept 100% of requests from top-5K domains, 50% from remaining domains. This preserves the highest-value traffic.

4 Overflow Queue: Shed requests are written to a local disk queue (bounded at 2 GB per pod). A background thread drains the overflow to Kafka when capacity recovers. If overflow fills, remaining requests are dropped with a counter metric.

5 Circuit Breaker: If Kafka is completely unreachable for >60 seconds, pods flip to "record to S3" mode, writing compressed Avro batches directly to S3. A recovery job replays these into Kafka once the cluster is healthy.

⚠️ Failure Scenario: Kafka Cluster Rolling Restart

During a rolling broker restart (e.g., for version upgrades), partition leaders migrate between brokers. For a 24-broker cluster with 512 partitions:

  • Each broker restart takes ~90 seconds (controlled shutdown + leader election)
  • During each restart, ~21 partitions (512/24) briefly have no leader
  • Producers experience 200-500ms of increased latency for affected partitions
  • Mitigation: Kafka's min.insync.replicas=2 ensures writes continue to replicas during leader migration. Producer retries absorb the transient failures. Total rolling restart of 24 brokers takes ~45 minutes with 30-second intervals between restarts.

Technology Decision: Crawl Framework

⚖️ Why Scrapy?

Problem: Crawl 200K domains reliably, respecting robots.txt, rate limits, and error handling, with clean text extraction. Throughput requirement: ~2,000 pages/minute sustained.

✅ Scrapy (Chosen)

  • Mature Python crawl framework with built-in robots.txt compliance
  • Async I/O (Twisted reactor): single spider handles 16+ concurrent requests
  • Middleware pipeline: pluggable text extraction, deduplication, rate limiting
  • 8 workers × 4 spiders = 32 concurrent spiders → ~2,000 pages/min
  • Integrates with our Python ML stack (TF-IDF, scikit-learn)
  • ScrapyD for distributed deployment and job management

❌ Rejected Alternatives

  • Apache Nutch: Designed for full-web crawls (billions of pages); massive overhead for our curated 200K domain list; Java-based, doesn't integrate with our Python ML stack
  • Selenium/Playwright (headless browser): Needed for JavaScript-rendered pages, but 10x slower and resource-intensive; only ~5% of our curated domains require JS rendering, which we handle as a special case
  • Common Crawl (pre-crawled data): Free but stale (1-3 months old), incomplete coverage of our specific domain list, and we have no control over crawl frequency or page selection
  • Custom Go/Rust crawler: Higher throughput per node, but Scrapy's middleware ecosystem (robots.txt, retry, dedup) saves months of development; our crawl volume doesn't justify custom infrastructure

Text Extraction Pipeline: From HTML to Clean Text

Raw HTML is useless for TF-IDF vectorization. We need clean, readable text that represents the domain's actual content. Our extraction pipeline processes each fetched page through five stages:

1 HTML Fetch: Download the page with a 30-second timeout. Follow up to 3 redirects. Store the raw HTML hash for change detection on future crawls.

2 Boilerplate Removal: Use the readability-lxml library to extract the main content area, stripping navigation, sidebars, footers, and ad containers. This algorithm identifies the densest text block in the DOM tree.

3 Tag Stripping: Use BeautifulSoup to remove all remaining HTML tags, scripts, styles, and non-text elements. Convert HTML entities (&amp; → &). Normalize whitespace.

4 Language Detection: Run the extracted text through fastText's language identification model. Keep only English-language content (our classifier is English-only in v1). Non-English pages are flagged for future multilingual expansion.

5 Deduplication: Compute a SimHash of the extracted text. If the hash matches the previous crawl within a Hamming distance of 3, mark the domain as "unchanged" and skip re-classification. This saves ~40% of classification compute on stable domains.

🛡️ 1.5 Data Quality: The Filtering Gauntlet

Raw bidstream data is a minefield. At 50 billion bid requests per day, even a 1% error rate means 500 million bad records polluting our user score arrays. Every invalid event that slips through degrades audience segment quality and erodes the CPM premiums our segments command. The quality filter stage sits between the raw Kafka topic and the filtered topic, processing every event through a series of validation checks.

The Layman's Analogy

Airport security screening is the closest analogy. Every passenger (bid request) must pass through multiple checkpoints. Some are obviously fake (bot traffic — like someone wearing a rubber mask). Some have invalid passports (missing user IDs). Some are not allowed to fly today (no GDPR consent). Some are trying to sneak in through a back door (domain spoofing). Only passengers who pass ALL checkpoints board the plane (reach the filtered topic).

Gate-by-Gate Deep Dive

Gate 2: Bot & Invalid Traffic (IVT) Detection

This is our most impactful filter, removing ~35% of all traffic. Bot traffic inflates impressions, wastes compute, and — most critically for us — creates false user behavior signals that corrupt audience segments. If a bot "visits" travel websites 10,000 times, it should not be classified as a travel enthusiast.

Detection layers:

  • IAB/TAG Bot List (General IVT): Maintained by the industry consortium; contains ~2,000 known bot User-Agent signatures. Updated weekly. Catches crawlers (Googlebot, Bingbot), SEO tools, monitoring services. This alone catches ~15% of traffic.
  • IP Reputation Database: Purchased from threat intelligence providers (e.g., Digital Element, MaxMind). Flags datacenter IPs, known proxy/VPN exits, and TOR exit nodes. Datacenter traffic accounts for ~12% of bid requests and is almost entirely non-human.
  • Behavioral Heuristics (Sophisticated IVT): Flagged by rate anomalies: a single user_id generating >100 bid requests/minute across >50 unique domains is almost certainly a bot. Similarly, identical device fingerprints appearing across multiple geo regions simultaneously indicate spoofed requests.
  • ads.txt / sellers.json Validation: Verify that the SSP sending the bid request is authorized to sell inventory for the declared domain. If cnn.com's ads.txt does not list the SSP as an authorized seller, the request is likely spoofed inventory.

🕑 Worked Example: Bot Detection in Action

At 2:00 AM EST, our pipeline sees a spike of bid requests with these characteristics:

  • User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (looks legitimate)
  • device.ifa: A1B2C3D4-E5F6-7890-ABCD-EF1234567890
  • 400 bid requests in the last 60 seconds, across 180 unique domains
  • All from IP range 45.33.32.0/24 (known Linode datacenter block)

Gate 2 verdict:

  1. User-Agent: passes IAB bot list check (not a known bot signature)
  2. IP: FLAGGED — datacenter IP in reputation database
  3. Behavioral: FLAGGED — 400 req/min far exceeds 100 req/min threshold
  4. Result: REJECTED as Sophisticated IVT. Counter incremented: ivt.datacenter_ip +1, ivt.rate_anomaly +1

Without this filter, this single bot would have generated 400 false score updates, potentially pushing its (fake) user profile into multiple high-value audience segments.

Gate 4: Consent Validation (GDPR / CCPA / COPPA)

Privacy compliance is non-negotiable. Processing user data without proper consent exposes the company to fines of up to 4% of annual revenue under GDPR. Our consent validation parses the IAB Transparency and Consent Framework (TCF) v2 consent string.

TCF v2 Consent String Parsing

Consent string example:
  "CPceAAAOPceAAGGABBENDaCgAAAAAAAAAAAAAAAAAAAA.YAAAAAAAAAA"

Decoded fields relevant to us:
  - CMP ID: 14 (known, valid Consent Management Platform)
  - Purpose Consents bitfield:
      Purpose 1 (Store/Access device): ✓ GRANTED
      Purpose 3 (Create personalized ads): ✓ GRANTED
      Purpose 4 (Content personalization): ✓ GRANTED
  - Vendor Consents: Our vendor ID (hypothetical 542): ✓ GRANTED

Decision logic:
  IF regs.ext.gdpr == 1:
    IF Purpose 1 AND (Purpose 3 OR Purpose 4) AND our_vendor_id consented:
      → PASS (process user data)
    ELSE:
      → REJECT (do not store/update user score array)

  IF us_privacy string present (CCPA):
    Parse: "1YNN" → version=1, notice=Y, opt-out=N, LSPA=N
    IF opt-out == "Y":
      → REJECT (user opted out of sale of personal information)

  IF regs.coppa == 1:
    → REJECT (child-directed content, never process)

Gate 5: Domain Spoofing Detection

Domain spoofing is a pervasive ad fraud technique where a low-quality domain masquerades as a premium domain (e.g., a spam site claims to be nytimes.com) to command higher CPMs. For our system, spoofed domains are doubly dangerous: they not only waste ad spend, but they also inject incorrect IAB category signals into user profiles.

Our spoofing detection uses three signals:

  • ads.txt cross-reference: Check that the SSP sending the bid request is listed as an authorized seller in the domain's ads.txt file. We cache ads.txt files for all 200K curated domains (refreshed weekly). A domain-SSP mismatch is a strong spoofing signal.
  • sellers.json verification: The SSP's sellers.json file lists all publishers authorized to sell through that SSP. Cross-reference the claimed domain against this list.
  • Historical pattern check: If a domain suddenly appears from a new SSP that has never sent traffic for that domain before, and the volume is significant, flag for manual review.

Gate 6: Deduplication

A single page load can generate 3-8 bid requests (one per ad slot on the page). We do not want to count a single page visit as 8 separate browsing events. Our deduplication uses a Bloom filter keyed on (user_id, domain, minute_bucket).

Bloom Filter Sizing

Expected items per minute: 580,000 QPS × 60 sec = 34.8M items
Target false positive rate: 0.1% (1 in 1,000)
Window: 1 minute (rotate Bloom filter every minute)

Bloom filter size:
  m = -(n × ln(p)) / (ln(2))²
  m = -(34,800,000 × ln(0.001)) / (0.693)²
  m = -(34,800,000 × -6.908) / 0.480
  m = 500,700,000 bits ≈ 60 MB per filter

Number of hash functions:
  k = (m/n) × ln(2) = (500.7M / 34.8M) × 0.693 = 10 hash functions

Implementation: Two rotating Bloom filters (current minute + previous minute)
Memory: 2 × 60 MB = 120 MB per filter worker node

Quality Filter Performance Budget

The quality filter consumer group must keep pace with the 580K msg/sec production rate. Our latency budget per event:

Gate Operation Latency (P99) Implementation

💡 Key Insight: All Quality Gates Are In-Memory

Every quality filter gate operates on in-memory data structures: hash maps, Bloom filters, precompiled regex, and locally-cached lookup tables. There are zero network calls in the per-event hot path. The lookup data (IP reputation database, bot list, ads.txt cache, known-fake IDs) is refreshed asynchronously in the background every 15-60 minutes, depending on the source. This design ensures that the quality filter never becomes a latency bottleneck, even at 580K QPS.

⚖️ Why Redis for Hot Tier?

✅ Redis Cluster (Chosen)

  • Sub-millisecond reads: P99 <2ms for point lookups
  • Native hash operations: HGET/HSET for individual score updates without reading entire object
  • Cluster mode: automatic sharding across 6+ primaries
  • Persistence options: RDB snapshots + AOF for durability
  • Atomic operations: HINCRBY for score accumulation without read-modify-write race conditions

❌ Rejected Alternatives

  • Aerospike: Excellent SSD-backed performance, but our active dataset (56 GB) fits comfortably in RAM; Aerospike's SSD advantage is unnecessary here and adds operational complexity
  • DynamoDB: Managed convenience, but cost at 95K writes/sec with on-demand pricing is ~$45K/month; also, P99 latency is 5-10ms vs Redis's <2ms
  • Memcached: Faster for pure reads, but no persistence, no atomic increment operations, no clustering with automatic resharding

Warm Tier Deep Dive: ScyllaDB for User Profiles

While Redis holds active users for real-time scoring, ScyllaDB serves as the persistent store for all 500M+ user profiles and the 7-day bid event history.

Why ScyllaDB over Apache Cassandra?

ScyllaDB is a C++ rewrite of Apache Cassandra, wire-compatible with the CQL protocol but with 5-10x better throughput per node due to its shard-per-core architecture. At our scale (95K writes/sec), this means we need ~15 ScyllaDB nodes versus ~60+ Cassandra nodes for equivalent performance — a significant operational and cost advantage.

Redis-to-ScyllaDB Writeback Pattern

Every score update in Redis is also asynchronously written to ScyllaDB. This ensures durability: if a Redis node fails, the user profile can be reconstructed from ScyllaDB. The writeback uses a batched async writer that accumulates updates for 100ms, then sends a batch CQL INSERT to ScyllaDB. This reduces write amplification from 95K individual writes/sec to ~10K batched writes/sec.

⚠️ Failure Scenario: Redis Node Failure

When a Redis primary shard fails:

  1. T+0s: Redis Sentinel detects primary unreachable (health check every 1s)
  2. T+5s: Sentinel promotes replica to primary (automatic failover)
  3. T+5-10s: Clients reconnect to new primary. During this window, ~500K score updates are buffered in the consuming application's in-memory queue.
  4. T+10s: Buffered updates are drained to new primary. Data loss window: last 1-2 seconds of writes that hadn't replicated to the promoted replica.
  5. Recovery: For the ~2 seconds of lost writes, the affected user profiles are slightly stale. The next bid request from those users will trigger a fresh score update. No manual intervention needed.

Worst case: Full Redis cluster failure (all shards). Recovery: warm-load all 120M active user profiles from ScyllaDB. At ScyllaDB's read throughput of 100K reads/sec, this takes ~20 minutes. During warm-up, score updates are queued in Kafka (retention: 24h), then replayed.

Monitoring throughput alone does not guarantee data correctness. We inject canary events — synthetic bid requests with known domains and user IDs — every 60 seconds across all Kafka partitions. These canary events flow through the entire pipeline (ingestion → quality filter → score update → Redis/ScyllaDB) and are verified at each stage.

1 Canary Injector: Produces a synthetic bid request with domain=canary-test.internal, user_id=canary-{partition_id} to each of the 512 Kafka partitions. (512 canary events / minute)

2 Quality Filter Pass-Through: Canary events are whitelisted by domain (bypass bot/spoofing checks) but still exercise consent validation and dedup paths.

3 Score Update Verification: After a canary event is processed, the verifier reads the canary user's score from Redis and ScyllaDB. It checks that the score was updated within expected latency.

4 End-to-End Latency Measurement: The verifier computes the time from injection to confirmed score update. This gives us a true end-to-end pipeline latency metric, more meaningful than per-component latencies.

Failure Scenario: Cascading Pipeline Failure

⚠️ Scenario: ScyllaDB Compaction Storm Causes Cascading Latency

This is a real failure mode we design against:

  1. T+0: ScyllaDB triggers major compaction on 3 nodes simultaneously (normal maintenance, but timing is unfortunate)
  2. T+30s: Write latency on affected nodes spikes from 5ms to 200ms. Async writeback from Redis begins buffering.
  3. T+2min: Writeback buffer reaches 500 MB. Score update consumers begin slowing down because their writeback calls are blocking (back-pressure).
  4. T+5min: Kafka consumer lag for score-update group exceeds 60 seconds. WARNING alert fires.
  5. T+8min: Consumer lag exceeds 300 seconds. CRITICAL alert pages on-call engineer.

Recovery strategy:

  • Score update consumer detects ScyllaDB latency >100ms and switches to fire-and-forget mode: writes to ScyllaDB become fully async with no blocking. Some writes may be lost, but Redis remains the source of truth.
  • Once compaction completes (~15-30 minutes), latency normalizes. A reconciliation job compares Redis and ScyllaDB for the affected time window and backfills any missing writes.
  • Prevention: Configure ScyllaDB compaction to run during low-traffic hours (3-6 AM). Stagger compaction across nodes using per-node scheduling.
