IAB Audience Generation System — Comprehensive Overview

A production ML system for classifying website domains into IAB content categories and generating high-value audience segments from real-time bidstream data in programmatic advertising.

🎯 Executive Summary

We are building an end-to-end ML system that transforms raw programmatic advertising bidstream data into monetizable audience segments. The system classifies website domains into the IAB Content Taxonomy v2.2 (~700 categories across 2 tiers), tracks user browsing patterns in real time, and assigns each user to high-confidence IAB audience segments that advertisers use for campaign targeting.

Why this matters commercially: Well-classified audience segments command 2–5x CPM premiums compared to untargeted inventory. An advertiser selling luxury travel packages will pay significantly more to reach users reliably classified into "IAB-20: Travel" than to show ads to a random audience. Our system turns raw bid request logs — which every ad exchange already has — into a revenue-generating data asset.

Core approach: Rather than classifying domains in real time (which would blow our latency budget), we pre-compute a domain-to-IAB lookup table using TF-IDF text classification on crawled website content. At bid time, we simply look up the domain, update the user's IAB score array, apply exponential time-decay, and export the user's top audience segments. The result is a system that processes 50 billion+ bid requests per day with sub-10ms user score lookups.

💡 Problem Statement: Why IAB Audience Generation Matters

The Layman Explanation

A billboard company can place ads randomly on highways and hope the right people drive by, or it can learn which drivers frequent specific routes and place relevant ads on exactly those highways. A luxury car brand wants their ad on the route used by high-income commuters, not on a rural road with no traffic.

Programmatic advertising works the same way, but digitally and at massive scale. Every time someone loads a webpage, an auction happens in milliseconds to decide which ad to show them. Advertisers want to bid higher on users who are likely interested in their products. If we can tell an advertiser "this user has been reading travel sites, finance sites, and luxury lifestyle content for the past two weeks," that user becomes far more valuable to a travel or financial services advertiser.

Our system's job is to build this understanding of each user's interests by watching which websites they visit, classifying those websites into standardized categories (the IAB Content Taxonomy), and maintaining a running profile of each user's affinities. The result: advertisers can target "users interested in Travel" or "users interested in Business & Finance" and pay a premium for the privilege.

The Technical Framing

The problem decomposes into four interconnected ML and engineering challenges:

1
Domain-Level Text Classification: Given the text content of a website domain (e.g., nytimes.com, expedia.com), classify it into one or more of ~700 IAB categories with calibrated probability scores. This is a multi-label text classification problem. We use TF-IDF vectorization + a classifier rather than keyword matching because context matters — the word "bank" on chase.com means finance, while "bank" on a geography site means riverbank.
2
Real-Time Score Accumulation: For each of ~500 million unique users, maintain a persistent IAB score array (a vector of ~700 floating-point values). Every time we see a user in the bidstream visiting a classified domain, we look up the domain's IAB scores and add them to the user's array. This must happen at 50B+ events/day with sub-10ms latency per lookup.
3
Temporal Decay Modeling: A user who visited expedia.com yesterday is more likely to be actively planning travel than a user who visited it 3 months ago. We apply exponential time-decay (score × e-λt) with category-specific decay rates — "Travel" intent fades faster (half-life ~5 days) than "Education" interest (half-life ~30 days). This ensures our audience segments reflect current user intent, not historical noise.
4
Audience Segment Assignment: From each user's decayed score array, extract the top-K categories that exceed a confidence threshold. Map these to named audience segments (e.g., "IAB Travel Enthusiast," "IAB Finance Intender") that downstream DSPs and advertisers can target in their campaigns. The segments must be large enough to be useful (>1M users for programmatic scale) but precise enough to command premiums (>60% true positive rate).

Why This Is Hard

⚠ Scale vs. Latency Trade-off: We process 50 billion bid requests per day (~580K QPS sustained, 2M+ QPS at peak). Every bid request triggers a user score lookup that must complete in under 10ms. We cannot run ML inference at bid time — hence the pre-computed domain lookup table architecture.
⚠ Category Ambiguity: Many domains span multiple IAB categories. CNN covers news, politics, entertainment, health, travel, and technology. A single domain classification must produce a probability distribution across categories, not a single label. This makes evaluation harder (traditional accuracy metrics are insufficient) and requires careful threshold tuning.
⚠ Identity Fragmentation: With cookie deprecation, IDFA restrictions, and GAID changes, the same physical user may appear under multiple identifiers (or no identifier at all). Our system must gracefully handle sparse user histories and supplement with contextual signals when user-level tracking is unavailable.
⚠ Domain Coverage vs. Quality: The web has millions of domains, but the top 200K domains cover ~85% of bid volume. We must choose between classifying everything (expensive, noisy) or curating a high-quality subset (practical, but with a coverage gap for long-tail domains).

⚖ Key Design Decisions

Every system of this scale involves hard trade-offs. Below we document the five most consequential design decisions, the alternatives we considered, and why we chose the approach we did. These are not academic exercises — each decision directly impacts latency, accuracy, cost, or operational complexity.

Decision 1: TF-IDF + Classifier vs. Keyword Vector Store vs. Deep Learning Embeddings

Problem: How do we classify a website domain into IAB categories given its text content?

Chosen approach: TF-IDF + Linear Classifier (LinearSVC or Logistic Regression)

✅ Why TF-IDF + Classifier Wins

  • Handles polysemy: "bank" classified correctly in context (financial vs. geographic) because the model sees surrounding terms
  • Fast training & inference: TF-IDF vectorization + LinearSVC trains on 200K documents in ~15 minutes on a single machine
  • Interpretable: Feature importance shows which terms drive each category — critical for debugging misclassifications and explaining to business stakeholders
  • Well-studied for multi-label text classification: Strong baseline that is hard to beat with more complex approaches on structured web content
  • Low operational cost: No GPU required. Model is a sparse matrix (~50MB). Batch inference on 200K domains takes ~10 minutes.
  • Calibrated probabilities: Logistic Regression naturally produces calibrated probability distributions across categories (Platt scaling for SVM)

❌ Rejected Alternative: Keyword Vector Store (Embedding Similarity)

  • Build a vector store of IAB category keyword lists, embed website text, find nearest category by cosine similarity
  • Rejected because: Pure keyword matching fails on ambiguous terms. "Python" matches both Tech and Pets. "Apple" matches Tech, Food, and Finance. Requires extensive manual keyword curation per category that does not scale to 700 categories.
  • No learned decision boundaries — relies on hand-crafted keyword lists that drift over time
  • Cosine similarity does not produce well-calibrated probability scores across categories

❌ Rejected Alternative: Deep Learning Embeddings (BERT/DistilBERT)

  • Fine-tune a transformer model on domain text for IAB classification
  • Rejected because: Marginal accuracy gains (+2-3%) do not justify 50x training cost and GPU inference requirements
  • Batch inference on 200K domains: ~10 min with TF-IDF+SVM vs. ~8 hours with BERT (even on GPU)
  • Model size: 50MB (TF-IDF) vs. 400MB+ (DistilBERT). Harder to version, deploy, and debug.
  • Kept as future upgrade path if accuracy plateaus below acceptable thresholds on long-tail domains

Decision 2: Pre-Computed Lookup Table vs. Real-Time Classification

Problem: When a bid request arrives with domain "expedia.com," how do we get its IAB categories fast enough?

Chosen approach: Pre-computed domain → IAB lookup table, loaded in-memory on every bid processing node

✅ Why Pre-Computed Wins

  • O(1) lookup: Hash map lookup in 0.5-2ms. No model inference on the critical path.
  • Deterministic latency: Lookup time does not vary with domain complexity, text length, or model load. p50 ≈ p99.
  • No GPU required at bid time: Bid processing nodes are CPU-only, commodity hardware. Keeps cost at ~$0.02/1M bid requests.
  • Table fits in memory: 200K domains × ~600 bytes = ~120MB. Trivially fits on any server.
  • Testable independently: The lookup table is a static artifact that can be validated, diffed, and rolled back without touching the real-time pipeline.

❌ Rejected Alternative: Real-Time Classification

  • Run TF-IDF + classifier inference on each bid request's domain
  • Rejected because: Even fast inference (5-10ms per domain) blows the latency budget when multiplied by 580K QPS. Would require caching anyway (which converges to our chosen approach).
  • Cold-start on unseen domains causes latency spikes
  • Requires ML model serving infrastructure (TensorFlow Serving, Triton) on the bid path — operational complexity for minimal benefit

❌ Rejected Alternative: External Classification API

  • Call a third-party API (e.g., IAB classification service) at bid time
  • Rejected because: Network latency (20-50ms) exceeds our entire budget. Rate limits would throttle at our QPS. Cost at 50B requests/day would be astronomical. Creates external dependency in the critical path.
What about unknown domains? When a bid request contains a domain not in our lookup table (~15% of traffic), we fall back to: (1) OpenRTB cat/sectioncat fields from the SSP, (2) URL-pattern heuristics (e.g., *.edu → Education), (3) parent domain lookup (e.g., sub.example.com → look up example.com). Unknown domains are queued for next-cycle crawling and classification.

Decision 3: Curated Domain Crawling vs. Crawl All Domains

Problem: The web has millions of domains. Which ones do we crawl and classify?

Chosen approach: Crawl a curated list of ~200K high-traffic domains that account for ~85% of bid volume

✅ Why Curated Crawling Wins

  • 85% coverage from 200K domains: The web follows a power law. The top 200K domains by bid volume cover 85% of all bid requests we see.
  • Quality control: We can manually validate classification accuracy for high-volume domains. Misclassifying cnn.com (millions of daily bid requests) is catastrophic; misclassifying a random blog (10 bid requests/day) barely matters.
  • Manageable crawl cost: 200K domains at ~5 pages each = 1M pages. Crawl completes in 4-8 hours. Full web crawl would take weeks and cost 100x more.
  • Freshness: We can re-crawl the full curated list weekly and high-traffic domains daily. Full web recrawl would always be stale.
  • Reduced spam/fraud exposure: Long-tail domains are disproportionately low-quality or spoofed. Limiting to known domains reduces garbage-in-garbage-out risk.

❌ Rejected Alternative: Crawl All Domains

  • Crawl every unique domain seen in the bidstream (~350K+ including long-tail)
  • Rejected because: Long-tail domains have sparse, low-quality content. Many are parked, redirect, or contain minimal classifiable text. The crawl cost scales linearly but the coverage gain is logarithmic (~350K domains covers only 92% vs 85% for 200K — a 7% gain for 75% more work).
  • Many long-tail domains block crawlers or serve different content to crawlers vs. users
  • Noise from misclassified long-tail domains degrades overall audience segment quality

Decision 4: Exponential Per-Event Decay vs. Fixed Periodic Batch Reduction

Problem: User interests change over time. How do we ensure our score arrays reflect current intent, not stale history?

Chosen approach: Exponential decay applied per-event at read time: score × e-λΔt

✅ Why Per-Event Exponential Decay Wins

  • Smooth, continuous decay: Scores decrease naturally with time elapsed since each contributing event, not in arbitrary batch-sized jumps
  • Category-specific half-lives: Travel intent fades in ~5 days (people book or abandon), Education interest persists for ~30 days (semester-length cycles). A single global reduction factor cannot capture this.
  • No separate batch job: Decay is computed lazily at read time. No need for a nightly job that sweeps all 500M active users.
  • Mathematically principled: Exponential decay models the empirical observation that recent events are exponentially more predictive than older events for intent signals.
  • Cheap to compute: One exp() call per category per user read — adds <0.5ms to the score lookup.

❌ Rejected Alternative: Fixed Periodic Batch Reduction

  • Run a nightly batch job that multiplies all user scores by a decay factor (e.g., 0.95)
  • Rejected because: Creates artificial "sawtooth" patterns — scores jump at event time and drop at batch time. A user checked right before the batch job looks very different from one checked right after.
  • Cannot differentiate decay rates by category without running separate batch jobs per category
  • Requires processing all 500M+ active users nightly, even those with no new activity (~60% of users on a given day)
  • Batch failures leave scores un-decayed, causing stale audience segments until the next successful run

Decision 5: Supplementing with OpenRTB Metadata

Problem: Our domain classifier covers ~85% of bid traffic. How do we handle the remaining 15% and improve confidence for classified domains?

Chosen approach: Use OpenRTB bid request fields (cat, sectioncat, bcat, content.url) as supplementary signals

✅ Why Supplementary Signals Help

  • Free data: OpenRTB bid requests already contain site.cat and site.sectioncat fields declared by the SSP/publisher. No additional cost to read them.
  • Covers unknown domains: For the ~15% of traffic from domains not in our lookup table, SSP-declared categories provide a usable (if noisy) fallback.
  • Page-level granularity: Our domain classifier produces domain-level scores. But content.url can distinguish cnn.com/travel from cnn.com/politics. OpenRTB metadata helps with this page-level signal.
  • Cross-validation: When our classifier and the SSP-declared category agree, confidence increases. When they disagree, it flags potential misclassification or domain spoofing.

⚠ Caveats and Limitations

  • Noisy and inconsistent: SSP-declared categories are self-reported by publishers. Accuracy varies wildly (some studies show 30-40% mismatch with actual content).
  • Often missing: Only ~40-60% of bid requests include the cat field. Cannot be the primary signal.
  • Gaming risk: Publishers may declare high-value categories (e.g., Finance) to attract higher bids, regardless of actual content.
  • Our approach: Weight SSP-declared signals at 0.3x relative to our own classifier output. Use disagreement as a monitoring signal for potential domain fraud.

🛠 Technology Stack Overview

Our technology choices are driven by three constraints: latency (sub-10ms for bid-time operations), scale (50B+ events/day), and operational simplicity (small ML/engineering team). We avoid over-engineering: if a proven, boring technology meets the requirement, we prefer it over a shiny alternative.

💰 Competitive Advantages: Cost Efficiency & Early Bid Wins

Beyond classification accuracy and system performance, our architecture delivers two powerful commercial advantages over traditional audience platforms in adtech: dramatically lower storage costs and an information latency edge that lets our DSP win bids earlier at lower prices.

Advantage 1: Compact Probability Vectors vs. Traditional Segment Storage

Traditional adtech audience platforms store explicit segment memberships per user — a list of segment IDs, often with metadata, timestamps, source tags, and redundant entries across overlapping segments. A single user might belong to dozens of segments, each stored as a separate record with associated metadata. This approach scales poorly: as the number of audience segments grows, storage per user grows linearly.

Our approach is fundamentally different. We store a single compact probability vector per user — approximately 700 floating-point values (one per IAB category). This vector is the generating function from which any audience segment can be derived on-the-fly, rather than storing the materialized output of every possible segment assignment.

Dimension Traditional Segment Storage Our Probability Vector Approach
Per-User Storage ~5-20 KB (segment IDs + metadata, grows with segment count) ~1.5-2 KB compressed (~5.6 KB uncompressed for 700 floats + timestamps)
Storage for 500M Active Users 2.5 – 10 TB ~0.75 – 1.2 TB (compressed)
Scaling with New Segments Linear — each new segment adds storage per qualifying user Constant — new segments are derived from the same probability vector, no additional storage
Write Amplification High — a single user event may update dozens of segment memberships Low — a single user event updates one probability vector (one write to the KV-store)
Estimated KV-Store Cost (Redis/Aerospike) $80K – $200K/month at scale $25K – $45K/month at scale
Key Insight — Store the Generating Function, Not the Output: By storing only the IAB probability vector (a compact numerical representation of user interests), we can derive any audience segment on demand without materializing and storing every segment membership. Adding 100 new audience segments for a new advertiser vertical costs us zero additional storage — we simply apply new threshold rules to the existing probability vectors. Traditional platforms would need to backfill segment memberships for hundreds of millions of users.

Advantage 2: Real-Time Scoring Enables Earlier, Cheaper Bid Wins

This is the more subtle and commercially powerful advantage. Because our probability vectors update in real time as users browse (within milliseconds of each bid request), our DSP detects intent signals before competitors whose audience platforms rely on batch-updated segment lists (typically refreshed every 1–24 hours).

Cookieless Future — How Our System Adapts

With third-party cookie deprecation in Chrome (ongoing), IDFA restrictions on iOS, and increasing user opt-out rates, our system must work in a world where 30-50% of users have no persistent identifier. Here is how we handle this:

1
Contextual Fallback: When no user ID is available, we cannot update a user-level score array. Instead, we provide contextual audience signals based solely on the current domain classification. "This page is about Travel" is less powerful than "This user is a Travel Enthusiast" but still valuable — contextual targeting commands 1.2-1.5x CPM premiums (vs. 2-5x for user-level segments).
2
Alternative ID Support: We integrate with privacy-compliant alternative IDs: UID2 (The Trade Desk), RampID (LiveRamp), ID5, and SharedID. These are consent-based, encrypted identifiers that allow cross-site user recognition without third-party cookies. Our system treats these as first-class user identifiers alongside cookies and device IDs.
3
Cohort-Based Scoring: For users with no identifier at all, we can assign them to cohorts based on observable signals (device type + geo + time-of-day + current domain). A cohort-level IAB score array (e.g., "iPhone users in NYC visiting news sites at 8am") provides a probabilistic audience signal without individual tracking.
4
Data Minimization: We store only IAB category scores and timestamps — not the raw URLs visited, page content, or any PII. The score array is a lossy aggregation: you cannot reconstruct the user's browsing history from their IAB scores. This is privacy-friendly by construction.

Technical Privacy Controls

⚠ Data Retention Policy: User score arrays are automatically deleted after 90 days of inactivity (no new bid events for that user). The weekly batch cleanup job removes expired profiles from all storage tiers. Score arrays for users who opt out via CCPA/GPC are deleted within 72 hours (fast path) via a dedicated Kafka consumer that processes deletion events with priority.
  • Encryption at rest: All user score arrays encrypted with AES-256 in Redis and Aerospike. S3 exports use server-side encryption (SSE-S3).
  • Encryption in transit: TLS 1.3 for all inter-service communication. Kafka configured with SASL_SSL.
  • Access control: Score arrays accessible only by the scoring service and segment generation service. No human access to individual user profiles without a logged, justified request.
  • Audit logging: Every score read/write is logged with timestamp, service identity, and consent status. Logs retained for 12 months for regulatory audit compliance.
  • Pseudonymization: Internal user IDs are hashed (SHA-256 with salt) versions of the original device/cookie IDs. We cannot reverse the hash to get the original identifier.
