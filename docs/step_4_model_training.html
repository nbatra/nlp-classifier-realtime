Step 4: Model Training

IAB Category Audience Generation System — Training TF-IDF + OvR Logistic Regression classifiers for domain-to-IAB mapping, decay parameter optimization, evaluation frameworks, and continuous learning infrastructure

🎯 Core Objective

Train a multi-label text classification system that maps 200K+ website domains to ~700 IAB Content Taxonomy v2.2 categories with high precision and well-calibrated probability scores. These domain-to-IAB mappings power the entire downstream pipeline — user score arrays, audience segments, and ultimately the programmatic targeting that commands 2-5x CPM premiums for well-classified audiences.

This step is the foundation of classification quality. A 1% improvement in domain classification precision propagates through hundreds of millions of user profiles and directly impacts revenue.

  • Input: Crawled text content from 200K+ domains, ground truth IAB category labels from multiple sources
  • Output: Trained TF-IDF vectorizer + 700 binary OvR Logistic Regression classifiers, calibrated confidence scores, per-category thresholds, optimized decay parameters
  • Retraining: Full retraining monthly, incremental updates weekly for newly crawled domains
  • Quality Bar: Macro F1 ≥ 0.72, Micro F1 ≥ 0.85, per-category precision ≥ 0.65 for all Tier-1 categories

📑 4.2 Training Data Preparation

The quality of our domain classifier is bounded by the quality of our training labels. In programmatic advertising, there is no single canonical source of ground truth for domain-to-IAB category mappings. Instead, we must assemble labels from multiple noisy sources, assess their reliability, and construct a training dataset that is both large enough to cover 700 categories and clean enough to produce a trustworthy model.

Conceptual Approach: Building a restaurant review database is analogous: the restaurant's own description (publisher self-declared), reviews from food critics (human annotators), aggregated user ratings (SSP-provided categories), and automated food-type detection from menu photos (semi-supervised bootstrapping). No single source is perfect, but combining them intelligently gives you a high-quality dataset.

Source Priority and Conflict Resolution

When multiple sources disagree on a domain's IAB categories, we apply a weighted voting scheme:

🔢 Label Fusion Formula

For domain d and IAB category c, the fused label confidence is:


  confidence(d, c) = Σs ∈ sources ws × labels(d, c)


Where weights ws are:

  whuman = 1.0   (gold standard)

  wpublisher = 0.85   (high trust, publisher knows their content)

  wSSP = 0.60   (moderate trust, varies by SSP)

  wsemi-supervised = 0.40   (lowest trust, model-generated)


Final label(d, c) = 1 if confidence(d, c) ≥ threshold(c), else 0

Default threshold = 0.50, tuned per category on validation set

📝 Worked Example: Label Fusion for cnn.com

Consider labeling cnn.com for IAB category "News & Politics" (IAB-12):

  • Publisher self-declared: IAB-12 (News), IAB-12-1 (International News) → label = 1.0 for IAB-12
  • SSP-provided (Google AdX): IAB-12 (News) → label = 1.0 for IAB-12
  • SSP-provided (Pubmatic): IAB-12 (News), IAB-19 (Technology) → label = 1.0 for IAB-12
  • Human annotator: IAB-12 (News), IAB-3 (Business), IAB-11 (Law/Gov/Politics) → label = 1.0 for IAB-12

Fused confidence for IAB-12:

  = (1.0 × 1.0) + (0.85 × 1.0) + (0.60 × 1.0) + (0.60 × 1.0) + (0.40 × 0.0)

  = 1.0 + 0.85 + 0.60 + 0.60 + 0.0 = 3.05 (normalized: 3.05 / 3.45 = 0.88)

  ⇒ Label = 1 (well above 0.50 threshold)


Now consider IAB-19 (Technology) for cnn.com:

  = (1.0 × 0.0) + (0.85 × 0.0) + (0.60 × 0.0) + (0.60 × 1.0) + (0.40 × 0.0)

  = 0.60 (normalized: 0.60 / 3.45 = 0.17)

  ⇒ Label = 0 (below 0.50 threshold — only one SSP thinks it's tech)

Inter-Annotator Agreement

We maintain a pool of 3 annotators per domain for the human-labeled subset (~15K domains). We measure agreement using Cohen's kappa for pairwise agreement and Fleiss' kappa for multi-annotator agreement:

Category Tier Fleiss' Kappa Interpretation Example Categories
Tier-1 (29 categories) 0.82 Almost perfect agreement Sports, Automotive, Finance
Tier-2 — Clear subcategories 0.71 Substantial agreement IAB-17-18 "Football", IAB-4-5 "Credit Cards"

⚠️ Warning — Label Noise in SSP Categories: SSP-provided site.cat fields are notoriously noisy. Our analysis of 10,000 domains with both SSP and human labels found:
  • 32% of SSP labels use outdated IAB v1.0 category codes (must be mapped to v2.2)
  • 18% of SSP labels are incorrect at Tier-1 level (e.g., a cooking blog labeled as "Technology")
  • 45% of SSP labels are incomplete — they capture the primary category but miss secondary categories
  • 5% of SSP labels use non-standard or proprietary category codes

We apply SSP-specific trust weights based on measured accuracy per SSP. Google AdX labels get w=0.75; smaller SSPs get w=0.40.

Label Noise Handling

We use a three-pass approach to clean label noise:

1

Pass 1: Cross-Source Validation

If a domain has labels from 2+ sources and they all agree on a category, mark as high-confidence. If sources disagree, mark as needs-review. Domains with only one label source are marked low-confidence.

2

Pass 2: Confident Learning (Cleanlab)

Apply the Confident Learning framework to identify likely label errors. Train an initial model on all data, use out-of-fold predictions to estimate the joint distribution of noisy and true labels. Flag domains where the model's prediction strongly disagrees with the label (predicted probability < 0.1 for a positive label, or > 0.9 for a negative label).

Typically identifies ~3-5% of labels as likely errors, which are sent for human review.

3

Pass 3: Human Review of Flagged Labels

Send the ~8,000 flagged domains (from both cross-source disagreement and confident learning) to human annotators. Cost: ~$8,000 at $1/domain. This is a one-time investment that dramatically improves label quality.

4.2.3 Train / Validation / Test Split Strategy

With 200K labeled domains and extreme class imbalance (some categories have 50,000+ positive examples while others have fewer than 100), a naive random split would leave rare categories with insufficient validation and test examples.

🤔 Split Strategy Decision

Strategy Approach Pros Cons Decision
Simple random 70/15/15 Random shuffle and split Simple, fast Rare categories may have 0 validation examples Rejected
Stratified split (iterative) Iterative stratification preserving multi-label distribution Ensures every category has proportional representation in each split Complex implementation for multi-label ✅ Chosen
Time-based split Train on older crawls, test on newer Simulates real deployment conditions Newer domains may have different category distribution; not enough data for rare categories Used as secondary validation

We use the iterative stratification algorithm (Sechidis et al., 2011) which handles multi-label datasets by greedily assigning samples to splits while maintaining label proportions. The implementation from scikit-multilearn handles our 700-label space efficiently.

Training set: 70% = ~140,000 domains

Validation set: 15% = ~30,000 domains (used for threshold tuning, HPO, calibration)

Test set: 15% = ~30,000 domains (held out, never used during development)


Additional calibration holdout: 5,000 domains carved from validation set for Platt scaling / isotonic regression (Section 4.6)


Minimum per-category guarantee:

  Every IAB category with ≥20 positive examples gets at least 3 in validation and 3 in test

  Categories with <20 positive examples are flagged for data augmentation (Section 4.2.4)

4.2.4 Handling Rare IAB Categories (Data Augmentation)

The IAB taxonomy has an extreme long tail. Here is the category distribution from our labeled dataset:

Category Popularity Tier Number of Categories Domains per Category % of All Categories Examples
🟢 High volume ~50 5,000 - 80,000 7% News (IAB-12), Technology (IAB-19), Entertainment (IAB-1)
🟡 Medium volume ~150 500 - 5,000 21% Cooking (IAB-8-5), Running (IAB-17-28), Credit Cards (IAB-4-5)
🟠 Low volume ~300 50 - 500 43% Herbs for Cooking (IAB-8-8), Sailing (IAB-17-30), Tax Planning (IAB-4-11)
🔴 Very rare ~200 <50 29% Cigars (IAB-19-28), Power Boats (IAB-17-26), Scuba Diving (IAB-17-31)

Why This Matters: A binary classifier for "Cigars" (IAB-19-28) with only 23 positive examples and 200,000 negative examples will trivially predict "not cigars" for everything and achieve 99.99% accuracy. We need targeted strategies for the long tail.

Strategies for Rare Categories

1

Targeted Crawling for Rare Categories

For categories with <50 domains, we actively seek additional domains to crawl. Sources include:

  • Niche directory sites: Find domain lists for specific verticals (e.g., cigar review sites, scuba diving communities)
  • Search engine queries: Query for "best [category] websites" and crawl top results
  • Reddit/forum scraping: Find subreddits and forums about niche topics, extract linked domains

Target: Bring every category above 100 positive training examples. Cost: ~$2,000 in crawling infrastructure + 40 hours of analyst time.

2

Page-Level vs Domain-Level Training

For rare categories, we augment domain-level labels with page-level training data. A domain like nytimes.com has pages about cooking, travel, technology, and finance. Instead of using the domain as one training example with multi-label, we split into individual pages with more specific labels.

This converts 1 domain with 5 labels into 50+ pages with 1-2 labels each, dramatically increasing training data for specific categories.

3

Hierarchical Label Propagation

If a domain is labeled as Tier-2 category "IAB-17-28 Running/Jogging", it automatically gets the Tier-1 label "IAB-17 Sports". This means Tier-1 classifiers always have more training data than Tier-2 classifiers. For Tier-2 categories with very few examples, we can use the Tier-1 classifier as a pre-filter.

4

Minimum Viable Category Threshold

Categories with fewer than 30 training examples even after augmentation are marked as low-confidence classifiers. Their predictions are still generated but flagged in the lookup table so downstream systems can apply more conservative thresholds or fall back to Tier-1 predictions.

✅ Key Insight — Data Quality Over Data Quantity: For our 700-category multi-label problem, we found that having 100 clean, verified labels per category outperforms having 1,000 noisy labels. The label fusion and cleaning pipeline (Sections 4.2.1-4.2.2) is arguably more important than any model architecture choice.

⚙️ 4.3 TF-IDF + OvR Logistic Regression Training Pipeline

The end-to-end training pipeline takes raw website text and produces 700 calibrated binary classifiers — one for each IAB category — spanning text preprocessing, vocabulary fitting, TF-IDF matrix computation, hyperparameter optimization, and final model serialization.

Conceptual Approach: Building 700 specialized librarians — each an expert at detecting one specific genre. The "Sports Librarian" has learned which words (and word combinations) are most indicative of sports content. Given a new document, each librarian independently votes on whether it belongs to their genre, along with a confidence score. The TF-IDF step builds a standardized vocabulary that all 700 librarians share. The logistic regression training teaches each librarian which vocabulary words matter for their genre.

4.3.4 Logistic Regression Hyperparameter Tuning

Each of the 700 binary classifiers has its own set of hyperparameters. While we could tune each classifier independently, this would require 700 × (number of HP combinations) × 5 (CV folds) training runs. Instead, we use a two-stage strategy:

1

Stage 1: Global Hyperparameter Search (on top-50 categories)

Run a full grid search on the 50 most common IAB categories (which have the most training data and thus the most reliable CV estimates). Find the globally optimal hyperparameter configuration.

2

Stage 2: Per-Category Refinement (on problem categories)

For categories with Macro F1 < 0.60 after Stage 1, run a targeted search over a narrower range around the global optimum. This typically affects ~80 categories, mostly rare or ambiguous ones.

4.3.5 Training Infrastructure

🤔 Training Infrastructure Decision: Single Machine vs Distributed

✅ Single Machine (Chosen)

  • TF-IDF matrix fits in 128GB RAM
  • scikit-learn + joblib parallelism across 32 cores
  • No distributed system overhead (serialization, scheduling)
  • Full training in ~50 minutes
  • Simple debugging and reproducibility
  • Cost: ~$3/run on c5.9xlarge (36 vCPU, 72GB)

❌ Spark MLlib (Rejected)

  • Would need cluster setup for a 50-minute job
  • Spark MLlib LogReg doesn't support liblinear solver
  • Distributed TF-IDF adds serialization overhead
  • Cost: ~$15/run for 4-node cluster
  • Overkill for 200K documents × 500K features
  • Would be justified at 10M+ documents

Verdict: At 200K documents, a single beefy machine handles everything comfortably. We would revisit Spark MLlib only if our crawl corpus exceeds 2M domains or if TF-IDF matrix exceeds available RAM (unlikely with sparse representation).

🔍 4.5 Evaluation Framework

A domain classifier that assigns IAB categories to 200K+ domains must be rigorously evaluated — not just with a single aggregate metric, but with a comprehensive framework that reveals performance across the full spectrum of IAB categories, identifies systematic failure modes, and provides actionable guidance for improvement. This section defines our multi-level evaluation methodology.

4.5.1 Multi-Label Evaluation Metrics

Standard single-label metrics (accuracy, F1) are insufficient for our 700-category multi-label problem. We use a layered metric hierarchy:

Metric Formula / Definition What It Measures Target
Micro F1 Global TP, FP, FN across all categories Overall classification quality; dominated by frequent categories ≥ 0.85
Macro F1 Average F1 across all 700 categories (unweighted) Treats every category equally; reveals performance on rare categories ≥ 0.72
Weighted F1 F1 per category weighted by support Balanced view between micro and macro ≥ 0.83
Hamming Loss Fraction of wrong labels across all domains × categories Overall label error rate in the 200K × 700 label matrix ≤ 0.003
Subset Accuracy Fraction of domains where predicted label set exactly matches true set Strictest metric: all labels must match ≥ 0.55
Coverage Average number of predicted labels per domain Model's tendency to over/under-predict multi-label assignments 1.5 - 2.2 (matching ground truth avg of 1.8)
Ranking Loss Fraction of incorrectly ordered label pairs Quality of probability ranking (even before threshold) ≤ 0.05

🔢 Micro vs Macro F1: Why Both Matter

Micro F1 pools all predictions:

  Micro Precision = Σ TPc / (Σ TPc + Σ FPc)  for c = 1..700

  Micro Recall = Σ TPc / (Σ TPc + Σ FNc)  for c = 1..700

  Micro F1 = 2 × (Micro Precision × Micro Recall) / (Micro Precision + Micro Recall)


A model could achieve Micro F1 = 0.90 by being perfect on News, Entertainment, and Sports (which together account for 40% of all positive labels) while being terrible on 500 rare categories.


Macro F1 averages across categories:

  Macro F1 = (1/700) × Σ F1c  for c = 1..700


Macro F1 = 0.72 means the average category has F1 = 0.72, ensuring we do not ignore the long tail of rare but commercially important categories (niche segments like "Luxury Travel" or "Fine Dining" command premium CPMs).

1

🎯 4.6 Confidence Calibration Training

In our system, domain classification probabilities are not just used for binary yes/no decisions — they are directly accumulated into user score arrays. When a user visits espn.com, the IAB-17 "Sports" probability (say, 0.92) is added to that user's Sports affinity score. If the model outputs 0.92 but the true probability of correct classification is only 0.75, every user who visits ESPN gets an inflated Sports score, which cascades into audience segment quality.

Why This Matters: Poorly calibrated probabilities corrupt millions of user profiles. If our model is systematically overconfident (outputs 0.9 when true probability is 0.7), users accumulate inflated scores across all categories, making audience segments noisier and reducing the CPM premiums advertisers are willing to pay.

4.6.1 What Is Calibration?

Conceptual Approach: A weather forecaster who says "90% chance of rain" ten times is well-calibrated if it actually rains 9 out of those 10 times. If it only rains 6 times, the forecaster is overconfident. The classifier must be equally well-calibrated — when it says "0.8 probability of Sports", domains with that score should truly be Sports ~80% of the time.

🔢 Formal Calibration Definition

A classifier is perfectly calibrated if:

  P(Y = 1 | f(X) = p) = p    for all p ∈ [0, 1]


In practice, we measure calibration by binning predictions into buckets and comparing average predicted probability to actual positive fraction:

  For each bin bk = [pk, pk+1):

    avg_predicted = mean of predicted probabilities in bin

    actual_fraction = (# true positives in bin) / (# samples in bin)

    calibration_errork = |avg_predicted - actual_fraction|


Expected Calibration Error (ECE):

  ECE = Σk (|bk| / N) × |avg_predictedk - actual_fractionk|


Target: ECE ≤ 0.03 (within 3% of perfect calibration on average)

4.6.2 Calibration Methods Compared

🤔 Calibration Method Decision

Method How It Works Pros Cons Decision
Platt Scaling Fit a logistic regression on top of model outputs: P(y=1|f) = 1/(1+exp(-af-b)) Parametric; works well when miscalibration is monotonic; only 2 parameters to fit Assumes sigmoid shape of miscalibration; may not capture complex patterns ✅ Primary method
Isotonic Regression Fit a non-parametric monotonic function mapping model outputs to calibrated probs Non-parametric; can capture any monotonic miscalibration pattern Requires more calibration data; can overfit with small datasets Used for high-volume categories
Temperature Scaling Divide logits by learned temperature T before softmax Single parameter; originally designed for neural networks Designed for softmax outputs, not OvR binary classifiers ❌ Rejected (not applicable to our architecture)
Beta Calibration Fit a Beta distribution to model outputs 3 parameters; handles asymmetric miscalibration More complex; marginal improvement over Platt for LogReg ❌ Rejected (over-engineering)

4.6.3 Calibration Training Process

1

Carve Out Calibration Holdout Set

From the 30K validation set, hold out 5,000 domains exclusively for calibration training. These domains are never seen during classifier training or threshold tuning — this prevents double-dipping that could make calibration appear artificially good.

2

Generate Raw Predictions

Run all 700 classifiers on the 5,000 calibration domains. This produces a 5,000 × 700 matrix of raw predicted probabilities.

3

Fit Per-Category Calibrators

For each IAB category, fit a calibration model:

  • High-volume categories (>500 positives in calibration set): Isotonic regression (enough data to fit non-parametric model)
  • Medium/low-volume categories: Platt scaling (only 2 parameters, robust with fewer data points)
  • Very rare categories (<10 positives in calibration set): No calibration — use raw LogReg probabilities, which are already reasonably calibrated for rare binary classifiers
4

Evaluate Calibration Quality

Compute ECE, Brier score, and reliability diagrams on the remaining 25K validation domains (not used for calibration training).

In the OvR framework, each binary classifier faces a different level of imbalance, from moderate (News: 52K positive vs 148K negative, ~1:3 ratio) to extreme (Cigars: 87 positive vs 199,913 negative, ~1:2,300 ratio).

For IAB-19-28 "Cigars" with 87 positive and 199,913 negative examples:

  wpositive = 200,000 / (2 × 87) = 1,149.4

  wnegative = 200,000 / (2 × 199,913) = 0.5002


Effect: Each positive "Cigars" example contributes ~2,298x more to the loss function than a negative example. This forces the classifier to actually learn the decision boundary rather than trivially predicting all negatives.


For IAB-12 "News" with 52,000 positive and 148,000 negative examples:

  wpositive = 200,000 / (2 × 52,000) = 1.923

  wnegative = 200,000 / (2 × 148,000) = 0.676


For common categories, the weighting is mild (~2.8:1), providing a slight boost to recall without sacrificing precision.

4.7.3 Why SMOTE Was Rejected

⚠️ SMOTE Failure Analysis:

SMOTE (Synthetic Minority Over-sampling Technique) generates synthetic positive examples by interpolating between existing positive examples in feature space. For our TF-IDF features, this has critical problems:

  • Sparse feature interpolation is unreliable: Interpolating between two sparse TF-IDF vectors creates dense vectors that do not correspond to any real document. A synthetic "Cigars" document might contain words from two completely different cigar sites that would never co-occur naturally.
  • Training time explosion: For the 180 categories with extreme imbalance, SMOTE generates hundreds of thousands of synthetic samples, increasing the effective training set from 200K to ~2M and increasing training time from 22 minutes to 3.5 hours.
  • Marginal accuracy improvement: Only +1.1% Macro F1 over class_weight='balanced', which achieves similar effect through loss weighting without any data generation.
  • Calibration degradation: Models trained on SMOTE-augmented data produce poorly calibrated probabilities because the synthetic examples distort the class distribution.

4.7.4 Minimum Sample Thresholds and Category Grouping

For categories with extremely few training examples, even balanced class weights cannot produce a reliable classifier. We apply a tiered approach:

✅ Key Insight — class_weight='balanced' Is the Right Default: The elegance of scikit-learn's balanced weighting is that it automatically adapts to each category's imbalance ratio. No manual tuning needed for 700 different classifiers. Combined with per-category threshold optimization (Section 4.5.4), this gives us the best precision-recall trade-off without the complexity of resampling techniques. The +8.2% Macro F1 improvement over no weighting (0.723 vs 0.641) is the single largest accuracy gain from any technique in our pipeline.

⏳ 4.8 Decay Parameter (λ) Optimization

User interests are not static. A user who researched travel destinations last month may have already booked their trip and no longer belongs in the "Travel" audience. Conversely, a user's interest in their sports team is persistent across seasons. Our exponential decay model captures this with category-specific decay rates (λ), but these rates must be learned from data, not guessed.

Conceptual Approach: Think of interest decay like the half-life of a radioactive element — different elements decay at different rates. "Travel Intent" has a short half-life (~7 days) because travel decisions are transient. "Sports Fandom" has a long half-life (~90 days) because team loyalty persists. We need to learn these half-lives for each IAB category from actual user behavior data.

4.8.1 Decay Model Recap

🔢 Exponential Decay Formula

For a user's IAB category score after time t since the last relevant domain visit:


  score(c, t) = score(c, 0) × e-λc × t


Where:

  score(c, 0) = accumulated IAB category score at time of last relevant visit

  λc = decay rate for category c (higher = faster decay)

  t = time elapsed in days


The half-life (time for score to halve) is: t1/2 = ln(2) / λc ≈ 0.693 / λc


Example: If λTravel = 0.099, then t1/2 = 0.693 / 0.099 = 7 days

Example: If λSports = 0.0077, then t1/2 = 0.693 / 0.0077 = 90 days

4.8.2 Training Data for Decay Optimization

We cannot directly observe "user interest decay" — it is a latent variable. Instead, we use proxy signals from downstream campaign performance to estimate optimal decay rates.

Data Source Signal Volume How It Helps
Historical campaign conversions Users who converted on category-specific ads ~2M conversion events / month across all categories Ground truth for "this user was actually interested in category c at time t"
Click-through rates by recency CTR on category-specific ads vs days since last category-relevant visit ~500M impressions / month with category labels Reveals how interest (measured by CTR) decays over time per category
Repeat visit patterns Users returning to same-category domains after N days ~50B bid requests / month Reveals natural interest persistence per category

4.8.3 Optimization Approach

For each IAB category, we optimize λ to maximize the audience quality metric: the CTR lift of users in the audience segment vs the general population.

1

Construct Historical Cohorts

For each IAB category c, identify users who visited category-c domains in the past. Group them by recency (days since last visit): 1d, 3d, 7d, 14d, 30d, 60d, 90d.

2

Measure Conversion/Click Rates by Recency

For each recency cohort, measure CTR on category-specific advertising. This gives us an empirical decay curve for each category.

3

Fit Exponential Decay Curve

For each category, fit λ by minimizing the squared error between the model (e-λt) and the observed normalized CTR decay curve. We use scipy's curve_fit with bounds [0.001, 1.0].

4

Validate with Holdout Campaign Data

Apply optimized λ values to compute user scores, build audience segments, and compare CTR lift against the holdout month's campaign data. Compare against baseline uniform λ.

4.8.6 Decay Parameter Retraining Schedule

Decay parameters are optimized quarterly, not monthly like the classifier itself. The reasons are:

  • Decay parameters require 60-90 days of behavioral data to estimate reliably
  • Interest decay patterns change slowly (it takes a cultural shift to change how long people care about sports)
  • Quarterly cadence aligns with seasonal adjustments
  • Cost: Each optimization run requires joining 500M+ impression records with conversion data — ~$200 in compute
✅ Key Insight — Decay Optimization Has the Highest Business Impact Per Effort: Category-specific λ optimization required one data scientist for two weeks and produces a permanent 29% improvement in audience CTR lift. This translates directly to higher CPM premiums: advertisers willingly pay more for fresher, more accurate audience segments. At 50B daily bid requests, even a 5% CPM improvement generates millions in annual revenue.

4.9.4 Website Content Drift: Recrawl Strategy

Domains change content over time. A tech blog might pivot to crypto. A news site might add a lifestyle section. We need to detect and respond to these changes.

1

Scheduled Recrawl (Monthly)

Recrawl the top 20,000 domains (by bid request volume) every month. These domains account for ~60% of all bidstream traffic, so keeping them fresh has the highest impact. Full recrawl of all 200K domains quarterly.

2

Triggered Recrawl (On Demand)

When monitoring detects a domain's classification diverging from OpenRTB metadata (e.g., SSP suddenly reports different site.cat for a domain), trigger an immediate recrawl and reclassification of that domain.

3

Content Change Detection

After recrawl, compute cosine similarity between the new TF-IDF vector and the stored vector. If similarity < 0.85, the domain's content has changed significantly — reclassify immediately.

From our data: ~2% of domains show >15% content change per month. About 0.3% undergo major topic shifts (similarity < 0.5).

4.11.2 Scenario 1: Training Data Poisoning

🔴 What Happens: An SSP begins sending incorrect site.cat labels for a large number of domains (e.g., labeling everything as IAB-19 "Technology" due to a bug in their taxonomy mapping). Since SSP labels feed into our label fusion pipeline, these incorrect labels contaminate our training data.

Detection

  • Label distribution monitoring: Alert when any single IAB category's label count changes by >20% between monthly snapshots
  • SSP-level label quality: Track per-SSP agreement rate with human labels; alert when a previously reliable SSP's agreement drops >10%
  • Cross-source disagreement spike: Alert when label fusion conflict rate exceeds 2x historical average

Recovery

1

Identify and Quarantine Affected Labels

Compare current labels against the previous month's snapshot. Identify domains whose labels changed from a single SSP source. Quarantine these labels (remove from training data) pending investigation.

2

Retrain Without Contaminated Source

Temporarily remove the suspect SSP from the label fusion pipeline and retrain. The label fusion system is designed to be resilient — removing one of four sources still produces usable labels for most domains.

3

Rollback If Already Deployed

If a contaminated model was already deployed, execute the rollback procedure (20 minutes). Roll back to the last known-good model version.

Scenario: SSP "AdColony" has a bug that maps all gaming app domains to IAB-1 "Arts & Entertainment" instead of IAB-9-30 "Video & Computer Games".

Impact: ~3,500 gaming domains get incorrect IAB-1 labels. The IAB-1 classifier learns spurious signals from gaming vocabulary; IAB-9-30 classifier loses training examples.

Detection: Label distribution alert fires when IAB-1 count jumps +8% and IAB-9-30 drops -15% in monthly snapshot.

Recovery time: Detection within 2 days of data snapshot. Label quarantine: 1 hour. Retrain without AdColony labels: 5 hours. Total: ~2 days from initial poisoning.

4.11.3 Scenario 2: Catastrophic Model Update

🔴 What Happens: Monthly retraining produces a model that passes automated QA gates (aggregate metrics look fine) but has a severe regression on specific important categories — e.g., IAB-4 "Careers" F1 drops from 0.83 to 0.45 due to a data preprocessing bug that corrupts job-related bigrams.

  • Shadow scoring (48h): Both champion and challenger score the same domains; per-category F1 comparison catches regressions that aggregate metrics hide

Recovery

  • Immediate: Rollback to previous model version (20 minutes)
  • Investigation: Compare model weights between versions; identify which features changed most for affected categories
  • Root cause fix: Fix the preprocessing bug, add regression test, retrain

4.11.5 Scenario 4: IAB Taxonomy Update

📅 What Happens: IAB Tech Lab releases IAB Content Taxonomy v3.0, adding 50 new categories, renaming 30, merging 15, and restructuring the hierarchy. This is the most disruptive event in our model's lifecycle — it happens rarely (every 1-2 years) but requires significant work.

Impact Assessment

  • New categories (50): Need new training labels, new classifiers — these will start with low accuracy until enough labeled data is collected
  • Renamed categories (30): Mostly cosmetic; update labels, retrain classifiers with same data
  • Merged categories (15): Combine training data from merged categories; retrain as single classifier
  • Restructured hierarchy: May affect Tier-1/Tier-2 relationships; update hierarchical label propagation rules

Migration Plan

1

Taxonomy Mapping Table (Week 1)

Create v2.2 → v3.0 mapping table. For each old category, map to the closest new category. This is a manual process requiring domain expertise — budget 3 days of analyst time.

2

Re-label Training Data (Week 1-2)

Apply the mapping table to convert all existing training labels from v2.2 to v3.0. For new categories with no v2.2 equivalent, initiate targeted crawling and human labeling.

3

Retrain Full Model (Week 2)

Full retraining with v3.0 labels. New categories will have limited data — use the strategies from Section 4.7 (class imbalance handling) aggressively.

4

Parallel Deployment (Week 3-4)

Run both v2.2 and v3.0 models in parallel for 2 weeks. Existing audience segments continue using v2.2 mappings while v3.0 segments are validated. Gradual cutover to v3.0 for new campaigns.

4.11.6 Graceful Degradation Principles

When failures occur, the system should degrade gracefully rather than fail completely:

Failure Graceful Degradation User Impact
Model retraining fails Continue serving previous model version indefinitely Slightly stale classifications; new domains remain unclassified
Specific category classifier fails Fall back to Tier-1 classification only for affected Tier-2 categories Less granular audience segments for affected categories
Calibration model fails Use raw LogReg probabilities (already reasonably calibrated) Slightly less precise user scores; ECE ~0.058 instead of ~0.024
Domain not in lookup table Use OpenRTB site.cat field from bid request as fallback Lower-quality classification but better than nothing
Training data snapshot unavailable Use previous month's snapshot + incremental updates only Model trained on slightly stale data; acceptable for one month
✅ Key Insight — The Lookup Table Is Your Safety Net: Because domain classification happens in batch (not real-time), the production system never directly depends on the training pipeline being healthy. The domain-to-IAB lookup table in Redis/Aerospike continues serving even if training has been broken for weeks. The worst case is that new domains are unclassified and existing domains may have stale classifications — but the system never goes down. This decoupling between training and serving is a key architectural advantage.
