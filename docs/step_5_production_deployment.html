Component Interaction: Real-Time Bid Processing Flow

When a bid request arrives from an SSP, the following sequence executes in under 10ms end-to-end:

1 Bid Request Ingestion (0.5ms): SSP sends OpenRTB bid request to our Kafka ingestion endpoint. The request contains device.ifa (device ID), site.domain (publisher domain), and optionally site.cat / bcat (IAB category hints). Kafka producer acknowledges within 0.5ms.

2 Kafka → Flink (1ms): Flink consumer picks up the event from the Kafka partition. The event is routed by user_id hash to ensure all events for the same user land on the same Flink task slot (key-by partitioning), enabling local state access.

3 Domain Lookup (2ms): Flink task queries Redis with the domain key (e.g., dom:cnn.com). Redis returns a sparse vector of (IAB_category_id, probability) pairs. For cnn.com, this might return: {IAB12: 0.85, IAB12-1: 0.72, IAB3: 0.15, IAB1: 0.12}.

4 OpenRTB Metadata Supplement (0.2ms): If the bid request includes site.cat (e.g., "IAB12" for News), we boost the corresponding category probability by a configurable factor (default: +0.1). If bcat excludes categories, we zero them out.

5 User Score Read + Decay (3ms): Read the user's current score array from Redis (keyed by usr:{device_id}). Apply exponential decay: score_i = score_i × e^(-λ_i × Δt) where Δt is time since last update and λ_i is category-specific decay rate. Travel (λ=0.15/day) decays faster than Education (λ=0.03/day).

6 Score Update + Write (2ms): Add the domain's IAB probability vector to the decayed score array: new_score_i = decayed_score_i + domain_prob_i. Write the updated array back to Redis with current timestamp. Asynchronously replicate to ScyllaDB.

7 Total End-to-End: ~8.7ms — well within our 10ms P99 budget.

Technology Decisions

Service Mesh: Envoy Proxy (via Istio) vs. Linkerd vs. No Service Mesh

Decision: Envoy Proxy (Istio) for service-to-service communication

✅ Why Envoy / Istio

  • Automatic mTLS between all services — critical for PII-adjacent user data
  • Traffic splitting for canary deployments (1% → 5% → 25% → 100%)
  • Circuit breaking prevents cascade failures from Redis or ScyllaDB outages
  • Distributed tracing (Jaeger integration) for end-to-end latency profiling
  • Retries with exponential backoff at the proxy layer — application code stays simple

❌ Rejected Alternatives

  • Linkerd: Lighter weight, but weaker traffic splitting capabilities — we need precise percentage-based routing for champion-challenger deployment
  • No service mesh: Would require implementing mTLS, circuit breaking, and traffic splitting in each service — too much operational burden at 7+ microservices
  • AWS App Mesh: AWS-only lock-in; we maintain multi-cloud optionality for cost leverage

API Gateway: Kong vs. AWS API Gateway vs. Envoy as Gateway

Decision: Kong API Gateway for external-facing segment serving API

✅ Why Kong

  • Plugin ecosystem for rate limiting, API key auth, and request transformation
  • Per-client rate limiting — different DSP partners have different QPS quotas
  • Built-in analytics for API usage tracking (which DSPs query which segments)
  • Can run in Kubernetes (Kong Ingress Controller) alongside our services

❌ Rejected Alternatives

  • AWS API Gateway: 29ms P99 added latency at high QPS — unacceptable when our total budget is 15ms
  • Envoy as API gateway: Requires custom Lua/WASM filters for API key auth and rate limiting — operational complexity not justified

Stream Processing: Apache Flink vs. Kafka Streams vs. Apache Spark Structured Streaming

Decision: Apache Flink for real-time bid event processing

✅ Why Flink

  • True event-time processing with watermarks — handles late-arriving bid events correctly
  • Exactly-once state semantics with RocksDB state backend + Kafka transactions
  • Async I/O operator for non-blocking Redis lookups — maximizes throughput per task slot
  • Savepoints enable zero-downtime deployments (stop with savepoint → deploy new version → resume)
  • Proven at similar scale: Alibaba processes 1B+ events/sec on Flink

❌ Rejected Alternatives

  • Kafka Streams: Simpler deployment (no separate cluster), but lacks async I/O — Redis lookups would block the processing thread, reducing per-instance throughput by 3-4x
  • Spark Structured Streaming: Micro-batch architecture adds 100-500ms latency — unacceptable for our <10ms requirement
  • Custom consumers: Maximum control but we'd have to build checkpointing, exactly-once, and backpressure from scratch
Why This Matters: The choice of Flink over Kafka Streams is not about capability — both can process 580K QPS. It is about latency under load. Flink's async I/O operator lets us issue Redis lookups without blocking the processing thread. At 580K QPS with 2ms Redis RTT, a synchronous approach would need 1,160 consumer threads just for Redis I/O. Flink's async approach handles this with ~75 TaskManagers, each running 3 task slots with 100-way async I/O concurrency.

🚀 5.3 Four-Phase Deployment Strategy

Deploying a new model version in a system that processes 50 billion bid requests per day is not a code push — it is a controlled experiment. A bad classifier update could misclassify millions of domains, corrupt user score arrays for hundreds of millions of users, and destroy audience segment quality that advertisers pay premium CPMs for. Our champion-challenger deployment strategy ensures every model version earns its way into production through four gates.

Why Not Blue-Green?

Challenge: Traditional blue-green deployment (flip 100% of traffic at once) is dangerous for ML systems because model quality issues are statistical, not binary. A new classifier might work perfectly on 95% of domains but catastrophically misclassify news domains as adult content — something only visible at scale. We need gradual exposure to catch distribution-level failures that unit tests and offline evaluation cannot surface.

Phase 1: Shadow Mode (Days 1-3)

🔮 Shadow Mode — Observe Without Impact

Traffic: 100% of bid events processed by both champion (current) and challenger (new) models. Only champion results are written to user profiles.

Goal: Compare challenger predictions against champion on live traffic without affecting any production outputs.

How It Works

  1. New domain lookup table (generated by challenger model) is loaded into a separate Redis keyspace (db=1 vs db=0 for champion)
  2. Flink scoring job is modified to perform dual lookups — one against champion (db=0), one against challenger (db=1)
  3. Only champion results update user score arrays. Challenger results are written to a shadow log topic in Kafka
  4. A comparison job reads both outputs and computes agreement metrics

Metrics Tracked in Shadow Mode

Metric Threshold to Pass What It Catches
Category agreement rate (top-1) ≥ 85% Catastrophic misclassification of common domains
Category agreement rate (top-3) ≥ 92% Subtle shifts in secondary categories
New uncovered domains ≥ 500 net new (improvement) Challenger should classify more domains, not fewer
P99 lookup latency (challenger Redis) ≤ 5ms (same as champion) Larger lookup table or different data structure causing slowdowns
Probability calibration (Brier score) ≤ champion + 0.02 Challenger must not be worse-calibrated than champion

Automated Gate:

If all metrics pass for 72 consecutive hours, an Airflow DAG automatically triggers Phase 2. If any metric breaches threshold, an alert fires and the challenger is held for investigation.

Phase 2: Canary Deployment (Days 4-7)

🐤 Canary — Small-Scale Real Impact

Traffic: 2% of users (randomly selected by user_id hash) receive score updates from the challenger model. 98% remain on champion.

Goal: Detect downstream impact on audience segment quality with real users, but limit blast radius.

How It Works

  1. Istio VirtualService routes 2% of Flink output to the challenger scoring path based on consistent hashing of user_id % 100 < 2
  2. Canary users' score arrays are updated using challenger domain probabilities
  3. Segment queries for canary users return segments computed from challenger-influenced scores
  4. We compare segment-level metrics between canary and control populations

Metrics Tracked in Canary

Metric Threshold to Pass What It Catches
Avg segments per user Within ±10% of champion Model producing too many or too few segments
Segment coverage (% users with ≥1 segment) ≥ champion - 2% Challenger not classifying enough domains = fewer user segments
Top-20 segment population sizes Within ±15% of champion Detecting category drift in popular segments
Segment stability (Jaccard similarity, 24h windows) ≥ 0.80 Users flipping in/out of segments too frequently
DSP bid rates on canary segments Within ±5% of champion If DSPs bid less on canary segments, quality may be worse

Automated Rollback Trigger:

If segment coverage drops by >3% or DSP bid rates drop by >8% over any 6-hour window, automatic rollback: Istio routes 100% back to champion, canary users' scores are re-initialized from ScyllaDB warm tier (pre-canary snapshot).

Phase 3: Gradual Rollout (Days 8-14)

📈 Gradual Rollout — Increasing Confidence

Traffic progression: 2% → 10% → 25% → 50% → 75% → 100%, with 24-hour holds at each stage.

Goal: Verify that positive canary results hold at increasing scale, catching issues that only appear with large user populations (e.g., long-tail domain coverage).

Rollback Criteria at Each Stage:

  • Automatic rollback: P99 latency exceeds 12ms (20% above budget), segment coverage drops >5%, or error rate exceeds 0.1%
  • Manual review gate: Triggered if any Tier-1 IAB category (e.g., IAB1-Arts, IAB12-News) population shifts by >20%
  • Business review gate at 50%: Advertising ops team reviews segment-level CPM trends before proceeding to 75%

Phase 4: Full Production (Day 14+)

✅ Full Production — Champion Promotion

Traffic: 100% on new model. Old champion model is decommissioned after a 7-day bake period.

Post-Deployment Actions:

  1. Promote challenger to champion in MLflow Model Registry (stage: "Production")
  2. Archive old champion artifacts to S3 Glacier (stage: "Archived")
  3. Update domain lookup table — swap Redis db=1 (challenger) to db=0 (new champion)
  4. Clear shadow infrastructure — remove dual-lookup code path from Flink job
  5. Generate deployment report — automated summary of all phase metrics, model diff, and approval chain
  6. Retain rollback capability — old champion's lookup table remains in S3 for 30 days for emergency rollback

📦 5.4 Model Export & Serving

Our IAB classification model is a two-stage pipeline: a TF-IDF vectorizer (150K+ vocabulary, producing sparse vectors) followed by 700 One-vs-Rest Logistic Regression classifiers (one per IAB Tier-2 category). Unlike deep learning models that need GPU serving infrastructure, our scikit-learn pipeline's primary serving path is batch scoring — we classify all 200K domains offline, then serve the results from a lookup table. The model itself never sees real-time traffic.

Why Batch Scoring (Not Real-Time Model Inference)?

Decision: Pre-computed lookup table over real-time model inference

✅ Why Batch + Lookup Table

  • Latency: Redis lookup = 2ms vs. TF-IDF + 700 LogReg inference = 50ms+ per domain
  • Cost: Score 200K domains once/week on spot instances ($3.50) vs. maintaining always-on inference fleet ($8,000/month)
  • Simplicity: No model serving framework needed (no TFServing, Triton, etc.)
  • Determinism: Every Flink worker sees exactly the same classification for a domain — no model version skew
  • Domain cardinality: 200K unique domains is small enough to pre-compute exhaustively

❌ When Batch Fails (Handled via Fallback)

  • New domains: A domain not in the lookup table gets no classification → fallback to OpenRTB site.cat field
  • Stale classifications: A domain changes content between weekly retrains → exponential decay naturally down-weights stale signals
  • Long-tail domains: Domains with <100 daily impressions may not be in our curated list → accepted trade-off (they contribute negligible signal)

Model Serialization: Joblib vs. Pickle vs. ONNX

Decision: Joblib for model serialization

Format Size (our model) Load Time Pros Cons Verdict
Joblib ~600 MB ~8 sec Native scikit-learn support, efficient numpy array serialization, compressed option Python-only, version coupling ✅ Chosen
Pickle ~750 MB ~12 sec Standard Python, no extra dependency Larger files (no numpy optimization), security risks with untrusted sources ❌ Larger, slower
ONNX ~400 MB ~3 sec Language-agnostic, optimized runtime, smaller footprint scikit-learn→ONNX conversion is lossy for sparse TF-IDF; OvR with 700 classifiers not well supported ❌ Conversion issues

Why Joblib wins: Since we only load the model in batch scoring jobs (not real-time serving), load time is not critical. Joblib's native scikit-learn integration means zero conversion overhead and guaranteed round-trip fidelity. The 600MB compressed artifact is stored in S3 and loaded once per batch run.

⚡ 5.5 Real-Time Scoring Infrastructure

A postal sorting facility processing 580,000 letters per second is the apt analogy: each letter (bid request) arrives with a return address (domain), must be looked up in a directory (domain lookup table), and the result stamped onto the recipient's file (user score array). Exactly-Once vs. At-Least-Once Semantics

Decision: At-Least-Once with Idempotent Updates

✅ Why At-Least-Once

  • Score updates are additive: Adding the same domain probability twice to a user score array is a minor overcount, not a catastrophic error. With 50B events/day and 500M users, average events/user/day = 100; a duplicate adds ~1% noise
  • Throughput gain: Exactly-once requires Kafka transactions + 2-phase commit with Redis, reducing throughput by ~40%
  • Checkpoint overhead: Exactly-once requires aligned barriers, causing back-pressure spikes at 580K QPS
  • Business tolerance: Audience segments use percentile-based normalization — small score noise is washed out in the normalization step

❌ Why Not Exactly-Once

  • Kafka EOS: Requires idempotent producer + transactional consumer → 40% throughput penalty at our scale
  • Redis transactions: MULTI/EXEC blocks are not compatible with pipelined writes, forcing serial execution
  • Complexity: 2PC between Flink state, Kafka offsets, and Redis is operationally fragile at 580K QPS

Safeguard: We deduplicate at the Kafka consumer level using a Bloom filter (10M entries, 0.1% FPR) keyed on (user_id, domain, timestamp_bucket_1min). This catches >99% of duplicates from Kafka rebalances while adding only 0.3ms latency.

Backpressure Handling at 580K QPS

1 Flink Native Backpressure: Flink's credit-based flow control automatically slows upstream operators when a downstream operator (e.g., Redis sink) falls behind. No data is lost — events buffer in network channels.

2 Kafka Consumer Lag Monitoring: If consumer lag exceeds 100,000 events (17 seconds at steady state), auto-scaling triggers additional Flink TaskManagers. Scaling action: add 10 TMs (30 task slots) per trigger, up to max 256.

3 Redis Circuit Breaker: If Redis P99 latency exceeds 8ms, the async domain lookup operator trips a circuit breaker. Events proceed with OpenRTB fallback categories (from site.cat). Circuit re-closes after 10 successful probes over 30 seconds.

4 Kafka Partition Overflow: Kafka brokers are configured with log.retention.bytes=50GB per partition. At 580K QPS × 500 bytes/event, each of 256 partitions fills at ~1.1 MB/sec = ~95 GB/day. With 50GB retention, we have ~12 hours of buffer before oldest events are dropped — providing ample recovery time for any consumer outage.

PII Handling and Data Masking

  • Device IDs in logs: All logging pipelines hash device IDs with SHA-256 + salt before writing to log storage. Raw IDs never appear in application logs, Datadog, or Grafana dashboards.

🔥 5.10 Failure Scenarios & Graceful Degradation

Every component in our system will fail. The question is not if, but how we degrade gracefully when it happens. Our design principle: stale data is better than no data, and no data is better than wrong data. Each failure mode has a detection mechanism, an automatic mitigation, and a recovery procedure.

Scenario 1: Redis Cluster Failure (Domain Lookup)

Failure: Redis cluster serving domain → IAB lookup table becomes unavailable (network partition, all nodes crash, or cluster enters FAIL state).

Detection → Mitigation → Recovery

DETECTION (within 5 seconds):
  - Flink AsyncDomainLookup operator: Redis timeout rate exceeds 5%
  - Datadog alert: redis.domain.error_rate > 0.05 for 5 consecutive checks
  - Circuit breaker trips after 10 consecutive failures (Resilience4j)

MITIGATION (immediate, automatic):
  1. Circuit breaker OPEN → all domain lookups bypass Redis
  2. Fallback chain activates:
     a. OpenRTB site.cat field → use SSP-provided IAB categories
        (available in ~60% of bid requests, accuracy ~70%)
     b. If no site.cat → check domain suffix heuristics:
        .edu → IAB5 (Education), .gov → IAB11 (Law/Gov)
     c. If no heuristic match → skip score update for this event
        (user profile unchanged, no harm done)
  3. Kafka consumer lag increases slightly (fallback is faster than Redis)

IMPACT:
  - Audience quality degrades: ~70% accuracy vs ~85% with Redis
  - ~40% of events get no domain signal (no site.cat, no heuristic)
  - User scores become stale but don't become wrong
  - Segment queries still work (read existing user scores from Redis user cluster)
  - Duration of degradation: minutes to hours depending on failure mode

RECOVERY:
  1. If node failure: Redis Sentinel promotes replica (automatic, 15 sec)
  2. If cluster failure: restore from S3 RDB snapshot + reload lookup table
     → Airflow trigger: reload_domain_lookup_table DAG
     → Time: 3 minutes (download 40MB + RESTORE commands)
  3. Circuit breaker half-open: probe Redis every 10 sec
  4. When 5 consecutive probes succeed → circuit CLOSED → resume normal lookups
  5. Post-incident: verify domain lookup hit rate returns to >95%

Scenario 2: Kafka Consumer Lag Spike

Failure: Flink consumers cannot keep pace with incoming bid events. Consumer lag exceeds 1 million events (≈100 seconds of traffic), growing continuously.

Detection → Mitigation → Recovery

DETECTION (within 2 minutes):
  - Flink dashboard: consumer lag metric exceeds 100K threshold
  - Datadog alert: kafka.consumer.lag > 1,000,000
  - Flink backpressure indicator shows upstream operators at >80%

ROOT CAUSES (in order of likelihood):
  a. Redis user score cluster slow (GC pause, network) → Flink blocked on writes
  b. Flink TaskManager OOM (memory leak, state growth) → restart loop
  c. Kafka rebalance storm (frequent TM restarts) → stop-the-world
  d. Traffic spike beyond auto-scale capacity

MITIGATION (tiered):
  Tier 1 (automatic, within 5 min):
    - Auto-scaler adds 10 Flink TaskManagers per trigger
    - Flink redistributes key groups to new TMs
    - Max scale: 85 TMs (256 Kafka partitions / 3 slots per TM)

  Tier 2 (if lag still growing after 10 min):
    - Enable "fast path" mode: skip ScyllaDB write sink
      (Redis-only writes, accept reduced durability)
    - Reduces per-event processing from 8ms to 5ms → 60% throughput gain

  Tier 3 (if lag exceeds 10M events / 30 min):
    - Activate sampling: process 1-in-N events (N=2 at 10M, N=5 at 50M)
    - Ensures we keep up with freshest events, sacrifice some history
    - Sampling is user-consistent (same hash as canary routing)

RECOVERY:
  - Once lag clears (< 50K events), disable sampling and fast-path
  - Scale down Flink TMs gradually (15 min cooldown between reductions)
  - Post-incident: analyze root cause, adjust baseline TM count if needed
  - Kafka retention (24h) ensures no data loss even with hours of lag

Scenario 3: Model Serving Failure (Stale Lookup Table)

Failure: Weekly batch scoring pipeline fails — new domain lookup table is not generated. Current lookup table is now 7+ days stale.

Detection → Mitigation → Recovery

DETECTION:
  - Airflow DAG failure alert (immediate)
  - Monitoring: domain_lookup_table.age_hours > 168 (7 days)
  - MLflow: no new model version promoted to "Staging" this week

IMPACT ASSESSMENT:
  - Existing 200K domains: classifications still valid (content drift
    is slow — websites don't change categories week-to-week)
  - New domains (est. 500-1,000/week): not classified → OpenRTB fallback
  - Measured quality degradation: ~0.3% audience accuracy loss per week
  - Acceptable for 1 missed cycle; concerning at 2+ missed cycles

MITIGATION:
  1. Redis TTL on domain keys is 14 days (2x retrain cadence)
     → Existing lookup table remains valid and served from Redis
  2. Alert on-call ML engineer for manual investigation
  3. Escalation timeline:
     - Day 7: Warning alert, auto-retry Airflow DAG
     - Day 10: Critical alert, notify ML team lead
     - Day 12: Emergency: force-deploy last successful model version
     - Day 14: Redis TTL starts expiring domains → coverage degrades

RECOVERY:
  1. Fix pipeline failure (common causes: crawler timeout, S3 permission,
     scikit-learn version incompatibility after infrastructure update)
  2. Trigger manual Airflow DAG run
  3. Verify new lookup table passes validation gates
  4. Push to Redis db=1 → run abbreviated shadow mode (24h)
  5. If shadow passes → swap to db=0 (skip full 14-day deployment cycle
     for pipeline fixes that don't change the model)

Scenario 4: Network Partition Between Services

Failure: Network partition isolates the Flink cluster from Redis user score cluster. Domain lookup Redis is still reachable (different AZ configuration).

Detection → Mitigation → Recovery

DETECTION (within 10 seconds):
  - Flink RedisScoreSink: connection timeout rate > 10%
  - Datadog: redis.user.connection.errors spikes
  - VPC Flow Logs: dropped packets between subnets

IMPACT:
  - Flink CAN read domain lookups (different Redis cluster, reachable)
  - Flink CANNOT write user score updates to Redis
  - Flink state (RocksDB) accumulates updates locally
  - Segment Service CAN read existing (stale) scores from Redis user cluster
  - Users get score updates, but Segment Service serves stale scores

MITIGATION (automatic):
  1. Flink score sink switches to "buffer mode":
     - Write score updates to RocksDB state only (already happening)
     - Queue Redis writes in a bounded buffer (100K entries max)
     - When buffer full, oldest entries evicted (LRU)
  2. Segment Service: detect freshness degradation
     - Add "X-Score-Freshness: degraded" header to API responses
     - DSPs informed that scores may be stale

RECOVERY (when partition heals):
  1. Flink drains buffered writes to Redis (burst of ~100K HMSET ops)
  2. For evicted entries: next user event triggers full score recalculation
  3. Freshness header returns to normal within 1 event cycle per user
  4. Post-incident: review VPC security group rules, consider dedicated
     ENI for Flink-to-Redis traffic

Scenario 5: Catastrophic Model Regression

Failure: A model update passes all deployment gates but gradually corrupts user scores over 48 hours due to a subtle calibration error — probabilities for IAB20 (Travel) are systematically inflated by 0.3, causing 60% of users to be incorrectly assigned to Travel audience segments.

Detection → Mitigation → Recovery

DETECTION (slow — this is the dangerous scenario):
  - Day 1: Normal monitoring shows Travel segment growing 40% (within
    the ±50% alerting threshold for gradual rollout — does not trigger)
  - Day 2: DSP partner reports anomalous bid rates on Travel segments
    ("Travel inventory doubled overnight, seems wrong")
  - Day 2: On-call investigates. Compares Travel segment population
    pre/post model update. Confirms: 60% of users now in IAB20 vs 22% baseline

MITIGATION (manual, high-priority):
  1. IMMEDIATE: Roll back to previous champion (v46) lookup table
     → Airflow DAG: emergency_rollback_domain_table
     → Copies v46 backup from S3 to Redis db=0 (2 minutes)
  2. IMMEDIATE: Halt current Flink scoring job (savepoint)
  3. Deploy Flink job with "score correction" mode:
     → For all users updated since model v47 went live:
       - Read user score array
       - Subtract v47 Travel inflation (estimated from shadow comparison)
       - Write corrected scores
     → Runs as batch job over ScyllaDB warm tier (~4 hours for 500M users)
  4. Resume normal scoring with v46 lookup table

RECOVERY (1-2 weeks):
  1. Root cause analysis: why did calibration error pass Platt scaling
     validation and shadow mode?
  2. Add detection: monitor per-category segment population growth rate
     → Alert if any Tier-1 category grows >25% in 24 hours
  3. Add gate: per-category calibration check during shadow mode
     → Compare probability distributions, not just agreement rates
  4. Retrain v48 with calibration fix
  5. Deploy v48 through full 14-day cycle (no shortcuts after regression)

BUSINESS IMPACT:
  - ~48 hours of inflated Travel audience → some DSPs overbid on Travel
  - Estimated advertiser waste: ~$50K (Travel CPM paid for non-Travel users)
  - Trust impact: 2 DSP partners request post-incident review
  - Remediation: credit affected DSPs, publish incident report
Key Insight: The most dangerous failures are slow, statistical degradations that pass threshold-based alerts. Scenario 5 illustrates why monitoring must include distribution-level metrics (segment population distributions, per-category probability distributions), not just aggregate accuracy. A 0.3 probability inflation on a single category looks fine in macro-F1 but catastrophically distorts audience segments.
