Step 3: Model Architecture - Website Classification & User Scoring

🎯 Core Objective

Design the model architecture that powers the two core intelligence layers of our IAB Audience Generation system: (1) a domain classification model that maps website domains to IAB content categories using TF-IDF text features, and (2) a user scoring model that accumulates, decays, and normalizes per-user IAB affinity scores from real-time bidstream data for programmatic audience targeting.

  • Domain Classification: Crawled website text → TF-IDF vectors → Multi-label classifier → IAB probability distribution per domain
  • User Scoring: Bid request stream → Domain lookup → Score accumulation with exponential decay → Normalized score array per user
  • Audience Assignment: Decayed score arrays → Top-K category selection → IAB audience segment membership

Together, these models serve 50 billion+ bid requests per day across 500M+ unique users, classifying 200K+ domains into the IAB Content Taxonomy v2.2 with its ~700 categories, enabling audience segments that command 2-5x CPM premiums for advertisers.

🤖 3.2 Domain Classification Model Architecture

Conceptual Approach: A librarian categorizing 200,000 websites into ~700 subject categories cannot read every page manually. Instead, a systematic process crawls each website, extracts representative text, converts that text into a numerical fingerprint (TF-IDF vector), and feeds that fingerprint into a trained classifier that outputs probability scores for each category. The result is a massive reference card catalog — look up any domain and instantly know its category distribution.

3.2.2 Text Preprocessing Details

The quality of our domain classification is only as good as the text we feed into the TF-IDF vectorizer. Crawled web content is notoriously noisy — filled with navigation menus, ad scripts, cookie consent banners, and boilerplate footers. Our preprocessing pipeline addresses each noise source.

1

HTML Stripping & Main Content Extraction

We use Mozilla's Readability algorithm (the same engine behind Firefox Reader View) to extract the main content block from each page, discarding navigation, sidebars, footers, and ad containers. For multi-page sites, we crawl up to 5 representative pages (homepage + top linked pages) and concatenate the extracted text.

2

Language Detection & Filtering

We run FastText's language identification model on the extracted text. Content in non-English languages is either (a) translated via a lightweight translation service for high-traffic domains, or (b) classified using language-specific models for the top-5 non-English languages (Spanish, French, German, Portuguese, Japanese). Domains with unrecognizable language or fewer than 50 words of extracted text are flagged for manual review.

3

Text Normalization

Lowercasing, removal of special characters and excessive whitespace, URL stripping, number normalization (e.g., "$19.99" → "PRICE_TOKEN"), and stopword removal using a custom AdTech-aware stopword list that retains industry-relevant terms like "mortgage", "insurance", "travel".

4

Domain-Level Text Aggregation

For each domain, we aggregate text from multiple crawled pages into a single document. We weight the homepage text at 2x (it typically contains the broadest category signals) and subpages at 1x. The aggregated document is truncated to 10,000 tokens to keep TF-IDF computation tractable.

3.2.3 TF-IDF Vectorization Configuration

The TF-IDF vectorizer is the bridge between raw text and machine-learning-ready feature vectors. Our configuration choices are informed by the characteristics of web content and the scale of the IAB taxonomy.

TF-IDF Configuration
TF-IDF Parameters: max_features = 50,000 (vocabulary size) ngram_range = (1, 2) (unigrams + bigrams) min_df = 5 (term must appear in at least 5 documents) max_df = 0.85 (ignore terms in >85% of documents) sublinear_tf = True (use 1 + log(tf) instead of raw tf) norm = 'l2' (L2-normalize each document vector) analyzer = 'word' (word-level tokenization) strip_accents = 'unicode' (normalize accented characters) Resulting Vector Space: - Each domain represented as a sparse vector in R^50,000 - Average non-zero entries per document: ~3,000-5,000 - Sparsity: ~92-96% - Storage: ~200K domains x ~4K avg non-zeros x 8 bytes = ~6.4 GB (sparse CSR format)

Why These Parameter Choices?

Parameter Value Rationale Alternative Considered

🧠 3.3 Classifier Selection: Why One-vs-Rest Logistic Regression?

Why This Matters: The classifier is the brain of our domain classification pipeline. It takes a 50,000-dimensional TF-IDF vector and outputs probability scores across ~700 IAB categories. The choice of classifier affects classification accuracy, calibration quality, training time, inference speed, and interpretability — all of which have downstream impacts on audience segment quality and CPM premiums.

3.3.1 Candidate Classifiers Evaluated

We evaluated five candidate classifiers against our requirements. The evaluation was conducted on a labeled dataset of 15,000 domains with ground-truth IAB category labels (human-annotated + IAB Tech Lab reference labels).

Decision: One-vs-Rest Logistic Regression

✅ Why Logistic Regression (OvR)

  • Probability calibration: Logistic regression outputs are naturally close to calibrated probabilities (sigmoid output), and with Platt scaling they become well-calibrated scores we can directly use as IAB affinity weights
  • Interpretability: Each category classifier has explicit feature weights — we can explain why a domain was classified as "Travel" by inspecting top TF-IDF features with highest weights (e.g., "hotel": +2.3, "booking": +1.9, "flight": +1.7)
  • Fast training: 12 minutes for ~700 binary classifiers on 200K domains. Enables weekly retraining without compute burden
  • Fast inference: 0.5ms per domain = 200K domains classified in ~100 seconds. Batch classification job completes in under 2 minutes
  • Sparse vector native: scikit-learn's SGDClassifier handles sparse TF-IDF matrices natively without densification, keeping memory under 8 GB
  • Well-understood regularization: L2 penalty prevents overfitting on rare categories; C parameter tuned via cross-validation
  • Debuggability in production: When a domain is misclassified, we can trace exactly which terms caused the error and add them to preprocessing rules

❌ Rejected Alternatives & Reasons

  • Linear SVM: Marginally better F1 (+1%), but outputs are decision function scores, not probabilities. Wrapping with Platt scaling adds complexity. For our use case, calibrated probabilities are essential since they become user score weights
  • Random Forest: Poor performance on high-dimensional sparse data (TF-IDF vectors are 50K-dim). 7% lower Micro-F1. Also 6x slower inference and much larger model size (~2 GB vs ~200 MB)
  • LightGBM: Requires dense features or custom sparse handling. Training is 7.5x slower. The 2% F1 gap vs LogReg doesn't justify the complexity, and tree-based models struggle with the very high dimensionality of TF-IDF space
  • Neural Network (MLP): Best raw accuracy (+3% Micro-F1), but 20x slower training, requires GPU, harder to debug misclassifications, and probability calibration requires additional temperature scaling. We may revisit this as a future improvement when accuracy becomes the bottleneck
Why Not Transformers / BERT? We deliberately chose TF-IDF + LogReg over transformer-based models (BERT, DistilBERT) for domain classification. While BERT would likely add 5-8% accuracy, the trade-offs are severe: (1) Inference cost: ~50ms per domain on GPU vs 0.5ms on CPU, increasing batch classification from 2 minutes to 3+ hours, (2) Infrastructure: Requires GPU instances for weekly batch job, adding ~$2,000/month in compute costs, (3) Model size: 400MB+ vs 200MB, (4) Debugging: Attention weights are hard to interpret for category misclassification root-cause analysis. For a pre-computed weekly batch job where the lookup table absorbs classification latency, the marginal accuracy gain doesn't justify 100x cost increase. We keep BERT as a future upgrade path if classification quality becomes the system bottleneck.

3.3.2 One-vs-Rest (OvR) Architecture for Multi-Label Classification

The IAB taxonomy has ~700 categories, and a single domain can belong to multiple categories. For example, "nytimes.com/travel" is both News (IAB12) and Travel (IAB20). This is a multi-label classification problem, not multi-class.

Our One-vs-Rest (OvR) architecture trains 700 independent binary classifiers, one per IAB category. Each classifier answers a simple yes/no question: "Does this domain belong to category X?" This naturally supports multi-label output because each classifier operates independently — multiple classifiers can fire "yes" for the same domain.

One-vs-Rest Architecture
Input: TF-IDF vector x ∈ R^50,000 For each IAB category c ∈ {1, 2, ..., 700}: Classifier_c: x → σ(w_c^T · x + b_c) = P(category=c | x) where: w_c ∈ R^50,000 = learned weight vector for category c b_c ∈ R = learned bias for category c σ(z) = 1/(1+e^(-z)) = sigmoid function Output: probability vector p = [P(c_1|x), P(c_2|x), ..., P(c_700|x)] Note: These are NOT constrained to sum to 1 (unlike softmax). A domain can have P(News)=0.82, P(Travel)=0.71, P(Arts)=0.03 → Assigned to both News AND Travel (both above threshold)

3.3.3 Handling Class Imbalance Across 700 Categories

The IAB taxonomy is heavily imbalanced. Broad categories like News (IAB12) and Technology & Computing (IAB19) have thousands of positive examples in our training set, while niche categories like Scuba Diving (IAB17-30) or Cigars (IAB9-6) may have only 20-50 positive examples.

Category Tier Example Positive Samples % of Training Set Challenge
High-frequency Tier-1 News (IAB12) ~8,500 4.25% Many sub-topics; classifier may over-generalize
Medium-frequency Tier-1 Travel (IAB20) ~3,200 1.6% Overlaps with hobbies, food, and culture
Low-frequency Tier-2 Scuba Diving (IAB17-30) ~45 0.02% Too few examples; prone to overfitting
Ambiguous Tier-2 Investing (IAB13-6) ~1,800 0.9% Overlaps with Personal Finance, Real Estate, Retirement

We address class imbalance through three mechanisms:

  1. class_weight='balanced' in SGDClassifier: Automatically adjusts the loss function weight inversely proportional to class frequency. For a category with 0.02% prevalence, positive samples are weighted ~5,000x more than negative samples, preventing the classifier from trivially predicting "not this category" for everything.
  2. Per-category threshold tuning: Instead of a global threshold of 0.5 for all categories, we optimize each category's threshold on a validation set to maximize F1 score. Rare categories typically end up with lower thresholds (0.15-0.30), while common categories use higher thresholds (0.40-0.60).
  3. Minimum confidence floor: Any category prediction below 0.10 is zeroed out regardless of threshold, preventing noise predictions from polluting the lookup table.
Pitfall: Threshold Tuning Overfitting. With 700 categories, tuning 700 individual thresholds on a small validation set risks overfitting the thresholds themselves. We mitigate this by grouping categories into 5 frequency bins and tuning one threshold per bin, then allowing per-category adjustments only for the top-50 most commercially valuable categories (those driving the most ad spend). This reduces the threshold search space from 700 to ~55, making cross-validation practical.

3.3.4 Hierarchical vs. Flat Classification

The IAB Content Taxonomy v2.2 is inherently hierarchical: 29 Tier-1 categories, each with 10-40 Tier-2 subcategories. Should our classifier exploit this hierarchy?

Decision: Flat Classification with Hierarchical Post-Processing

Option A: Flat Classifier (CHOSEN)

  • Single OvR model across all ~700 categories
  • Simpler training and serving pipeline
  • No error cascading: Tier-2 predictions are independent of Tier-1
  • Micro-F1: 0.78 (measured)
  • Training time: 12 min
  • Maintenance: 1 model, 1 retraining job

Option B: Hierarchical Classifier (REJECTED)

  • First classify Tier-1 (29 categories), then route to Tier-2 sub-classifier
  • Potentially better Tier-2 accuracy within parent categories
  • Error cascading: Wrong Tier-1 prediction means Tier-2 is always wrong
  • Micro-F1: 0.76 (measured — cascade errors hurt)
  • Training time: 45 min (29 sub-models + 1 parent)
  • Maintenance: 30 models, 30 retraining jobs

Rationale: We tested hierarchical classification and found that cascade errors reduced overall accuracy by 2%. When the Tier-1 classifier misclassifies a domain (e.g., predicts "Sports" instead of "Health & Fitness" for a yoga website), the Tier-2 sub-classifier for "Sports" has no chance of recovering the correct label. The flat classifier avoids this entirely. We still leverage the hierarchy in post-processing: after flat classification, we enforce consistency constraints (e.g., if a domain is classified as "Skiing" (IAB17-31) with high confidence, we boost its "Sports" (IAB17) Tier-1 score even if the flat classifier didn't assign it).

3.3.5 Confidence Calibration: Platt Scaling & Isotonic Regression

Why This Matters: The raw output of a logistic regression classifier is not always a well-calibrated probability. "Calibrated" means that when the model says P(Travel) = 0.70, then among all domains for which it predicts 0.70, approximately 70% truly belong to the Travel category. Calibration is critical because we use these probabilities directly as weights in user score accumulation — a miscalibrated 0.70 that should be 0.40 would inflate Travel scores by 75%.

Calibration Methods
Method 1: Platt Scaling - Fits a logistic regression on the classifier's raw outputs - P_calibrated = σ(A · f(x) + B) - where f(x) is the raw classifier score, A and B are learned on validation set - Works well when miscalibration is monotonic (sigmoid-shaped) Method 2: Isotonic Regression (CHOSEN) - Non-parametric monotonic regression on raw outputs - More flexible than Platt scaling — handles non-sigmoid miscalibration - Higher risk of overfitting on small validation sets - We use 5-fold CV to mitigate overfitting Calibration Quality (Brier Score, lower is better): Before calibration: 0.082 After Platt scaling: 0.054 After isotonic: 0.047 ← CHOSEN Reliability Diagram Bins (isotonic, post-calibration): Predicted | Actual | Gap 0.0 - 0.1 | 0.08 | 0.02 0.1 - 0.2 | 0.16 | 0.01 0.2 - 0.3 | 0.27 | 0.02 0.3 - 0.5 | 0.42 | 0.01 0.5 - 0.7 | 0.59 | 0.02 0.7 - 1.0 | 0.74 | 0.01
Key Insight: Isotonic regression reduces calibration error by 43% compared to uncalibrated outputs. This directly translates to better user scoring quality: when a domain is labeled P(Travel) = 0.60 after calibration, the user visiting that domain gets an appropriately weighted Travel score, not an inflated or deflated one. ⚠️ 3.5 Handling Edge Cases in Domain Classification

The clean pipeline described above works well for ~80% of domains. The remaining 20% present challenges that require special handling to prevent polluting the lookup table with low-quality classifications.

3.5.1 Multi-Topic Domains (reddit.com, medium.com, wikipedia.org)

Aggregator and user-generated content platforms don't have a single dominant topic. reddit.com hosts content about everything from r/investing to r/cooking. Classifying such domains at the domain level is fundamentally wrong — the content varies entirely by URL path.

1

Detection

We flag domains as "multi-topic" when the classifier's entropy over IAB categories exceeds a threshold. Formally: H(p) = -Σ p_c · log(p_c) > 3.5 (where 3.5 bits corresponds to roughly equal probability spread across ~11 categories). These domains are also flagged if they appear in our manually curated list of ~500 known aggregator domains.

2

URL-Path Level Classification

For flagged multi-topic domains, we attempt URL-path-level classification. Instead of classifying "reddit.com" as a single entity, we classify "reddit.com/r/personalfinance" and "reddit.com/r/travel" separately. This requires: (a) maintaining a sub-domain/path-level lookup table for the top-500 multi-topic domains, and (b) extracting the URL path from bid requests (available in the OpenRTB page field).

3

Fallback: Uniform Low-Confidence Scores

When URL-path classification isn't possible (no path available in bid request, or path not in sub-domain table), we assign uniform low-confidence scores across the top-5 broadest Tier-1 categories. This acts as a weak signal — better than nothing, but won't dramatically shift any user's score array.

3.5.2 Parked & Empty Domains

Approximately 5-8% of crawled domains return minimal content: parked domain pages ("This domain is for sale"), empty pages, or redirect chains. Our handling:

  • Detection: If extracted text is <50 tokens after preprocessing, flag as "insufficient content"
  • Action: Mark domain as "unclassified" in lookup table with a special sentinel value. Bid requests for unclassified domains fall through to the OpenRTB metadata fallback (using the bcat / cat fields from the bid request itself, which some SSPs populate based on their own classification)
  • Monitoring: Track the percentage of bid requests hitting unclassified domains. Alert if >5% of daily volume hits this fallback path

3.5.3 Foreign Language Content

Our primary classifier is trained on English text. For the ~15% of domains with non-English content:

  • Top-5 languages (Spanish, French, German, Portuguese, Japanese): We maintain separate TF-IDF + LogReg models trained on language-specific labeled datasets (~20K domains per language). These achieve ~72% Micro-F1, lower than English (78%) due to smaller training sets.
  • Other languages: We use a lightweight translation pipeline (MarianMT) to translate extracted text to English before classification. Translation adds ~2 seconds per domain, acceptable in batch processing. Quality is ~70% Micro-F1 after translation.
  • Mixed-language domains: Some domains have English navigation with non-English body content. We classify each language segment separately and merge probabilities using weighted averaging (weight proportional to text length in each language).

3.5.4 Newly Registered & Emerging Domains

Each week, ~2,000 new domains appear in our bidstream that aren't in the lookup table. These need classification before the next weekly batch run.

Fast-Path Classification for New Domains: We run a lightweight classification job daily (not weekly) for newly discovered domains. This job processes only the new domains (~2K/day vs 200K in the full weekly batch), completing in ~5 minutes. New domain classifications are promoted to the Redis lookup table immediately, with a "provisional" confidence flag that is upgraded to "confirmed" after the next full weekly batch run cross-validates the classification.

3.5.5 Domain Spoofing & Fraud

In programmatic advertising, domain spoofing is a significant fraud vector: a low-quality domain claims to be a premium domain (e.g., a pop-up ad site claims its traffic comes from nytimes.com). This affects our classification pipeline because:

  • The classified domain (nytimes.com) may not match the actual content being served
  • User score arrays could be polluted with incorrect category signals

Mitigations:

  1. ads.txt / sellers.json verification: Cross-reference the SSP selling the impression with the domain's ads.txt authorized seller list. Mismatches flag potential spoofing.
  2. Statistical anomaly detection: If a domain suddenly appears with 10x its normal bid volume or from unexpected geolocations, flag for review before trusting its classifications.
  3. OpenRTB bcat/cat cross-validation: Compare our classification with the SSP's reported categories. Significant disagreements trigger a spoofing alert.
Failure Scenario: Misclassified High-Traffic Domain. If cnn.com were accidentally classified as "Shopping" instead of "News", every user visiting CNN would accumulate Shopping scores. With CNN's ~50M monthly visitors, this could create a massive phantom "Shopping Intenders" audience segment, wasting advertiser spend and eroding trust. Mitigation: Top-1000 domains by bid volume are manually spot-checked after each weekly classification batch. Automated alerts fire if a top-1000 domain's primary category changes from the previous week.

👤 3.6 User Scoring Model Architecture

🎯 Part B: From Domain Visits to User Affinity Scores

While domain classification is a batch intelligence layer, user scoring is the real-time intelligence layer that runs on every bid request. When a user visits a domain, we look up that domain's IAB probability distribution and add those probabilities to the user's persistent score array. Over time, scores accumulate for frequently visited categories and decay for abandoned interests. The result is a 700-dimensional affinity profile per user that evolves with every impression.

Conceptual Approach: Think of each user as carrying an invisible scorecard with 700 slots — one for each IAB category. Every time the user visits a website, the librarian (our system) checks which categories that website belongs to and adds a tick mark to those slots. But the scorecard uses disappearing ink: older tick marks fade over time, so the scorecard always reflects the user's current interests, not what they browsed three months ago.

3.6.1 Score Accumulation Architecture

When a bid request arrives for user u on domain d at time t, the score update follows this flow:

Additive Score Accumulation: Why Not Multiplicative?

Decision: Additive Score Accumulation

✅ Additive: score[c] += p_c (CHOSEN)

  • Score grows linearly with visit frequency — 10 visits to Travel domains produces ~10x the score of 1 visit
  • New category signals always contribute, regardless of existing score
  • Simple, interpretable: "This user's Travel score of 4.7 means they have accumulated ~5 Travel-classified domain visits worth of affinity"
  • Works naturally with exponential decay — decay operates on the total score
  • Cold-start friendly: first visit immediately creates a non-zero signal

❌ Multiplicative: score[c] *= (1 + p_c) (REJECTED)

  • Score grows exponentially with visits — creates runaway scores for power users
  • Requires careful bounding to prevent overflow (scores can reach 10^6+ for heavy users)
  • First visit must initialize to 1.0 (not 0.0), creating a chicken-and-egg problem for cold-start
  • Harder to interpret: "Score of 47.3" is less meaningful than additive accumulation
  • Decay interaction is non-trivial: multiplicative decay on multiplicative accumulation creates complex dynamics

3.6.2 Exponential Decay Model

Why This Matters: User interests change over time. A user who researched Hawaii vacations last month but is now browsing mortgage rates should be in the "Personal Finance" segment, not the "Travel" segment. Exponential decay ensures that older signals lose influence smoothly, with the rate of decay tuned per category based on how quickly interests in that category typically change.

Exponential Decay Formulation
User Score for category c at current time t: S_c(t) = sum_{i=1}^{N} [ p_{c,i} * e^(-lambda_c * (t - t_i)) ] Where: N = total number of domain visits for this user p_{c,i} = IAB probability for category c from domain of visit i t_i = timestamp of visit i lambda_c = decay rate for category c (category-specific) t = current time Half-life relationship: t_{1/2} = ln(2) / lambda_c = 0.693 / lambda_c If lambda_c = 0.023/day -> half-life approx 30 days After 30 days, a score contribution is worth 50% of original After 60 days, worth 25% After 90 days, worth 12.5% Efficient computation (lazy evaluation): Instead of recomputing from all visits, we update incrementally: At time t, given last update at t_prev: dt = t - t_prev (in days) For each category c in user's score array: S_c(t) = S_c(t_prev) * e^(-lambda_c * dt) + p_{c,new} This reduces update from O(N_visits) to O(N_active_categories).

Category-Specific Decay Rates

Not all interests decay at the same rate. A user who researched "Travel" last week may have already booked their trip and moved on. A user who was reading "Education" content is likely in a sustained learning phase. We assign category-specific decay rates based on empirical analysis of purchase cycle duration and interest persistence:

IAB Category Decay Rate (lambda) Half-Life Rationale
Travel (IAB20) 0.023 /day ~30 days Travel intent is often short-lived: research, book, done. But trip planning can span 2-4 weeks.
Shopping (IAB22) 0.035 /day ~20 days Purchase intent decays rapidly post-purchase. "I just bought a laptop" means no more laptop ads needed.
Automotive (IAB2) 0.0077 /day ~90 days Car buying cycle is 3-6 months. Interest persists through research, dealership visits, financing.
Education (IAB5) 0.0038 /day ~180 days Educational interests are sustained: a student or professional learner maintains category engagement for months.
News (IAB12) 0.046 /day ~15 days News reading is habitual but non-specific. A News score should reflect recent behavior, not last month's.
Personal Finance (IAB13) 0.0116 /day ~60 days Financial research phases (mortgage, credit card, investment) typically span 1-3 months.
Real Estate (IAB21) 0.0058 /day ~120 days Home buying cycle is 3-6 months, similar to automotive but slightly longer.
Sports (IAB17) 0.023 /day ~30 days Sports fandom is persistent, but event-driven spikes (playoffs, tournaments) fade in ~4 weeks.
Health & Fitness (IAB7) 0.0116 /day ~60 days Health research often spans a treatment/condition lifecycle of 1-3 months.
Default (all others) 0.0154 /day ~45 days Default half-life of 45 days balances recency with stability for categories without specific tuning.
How We Tune Decay Rates: Decay rates are not guesses — they are calibrated using holdout advertiser conversion data. We run a grid search over lambda values for each major category, measuring which decay rate maximizes the correlation between a user's decayed score and their likelihood of converting (clicking/purchasing) on ads in that category. For example, for Travel, we found that lambda=0.023 (30-day half-life) produced the best conversion lift (3.2x) compared to faster decay (too aggressive, loses users still in planning phase) or slower decay (too lenient, includes users who already completed their trip).

3.6.3 Score Normalization

Raw accumulated scores are not directly comparable across categories. A user who visits 50 News domains per day will have a raw News score of ~40+ (since most news domains have P(News) ~ 0.80), while a user genuinely interested in Scuba Diving might have a raw Scuba Diving score of 0.8 after visiting 3 scuba sites. Without normalization, News would dominate every user's profile simply because news is the most browsed category.

Normalization Approach: Percentile-Based

We use percentile-based normalization, which ranks each user's category score against the population distribution for that category.

Percentile Normalization
For category c, user u with raw score S_c(u): normalized_score_c(u) = percentile_rank(S_c(u), distribution_c) Where distribution_c is the empirical distribution of raw scores for category c across all 500M users. Example: User's raw Travel score: 4.7 Travel score distribution: median=0.8, P75=2.1, P90=5.3, P95=8.0 User's percentile rank: ~85th percentile Normalized score: 0.85 User's raw News score: 38.2 News score distribution: median=12.0, P75=28.0, P90=45.0, P95=62.0 User's percentile rank: ~80th percentile Normalized score: 0.80 Result: Despite raw scores differing by 8x, the normalized scores correctly show this user has slightly stronger Travel affinity (85th) than News affinity (80th) relative to the population. Distribution updates: Recomputed weekly from a 10% random sample of active users (~50M users), stored as 100-bucket histograms per category in Redis for O(1) percentile lookup.

Decision: Percentile Normalization over Alternatives

✅ Percentile-Based (CHOSEN)

  • Category-agnostic comparison: a score of 0.85 in Travel and 0.85 in Finance means "top 15% in both"
  • Robust to category frequency differences (News vs Scuba Diving)
  • Robust to score inflation from high-traffic categories
  • Interpretable: "This user is in the 90th percentile for Travel affinity"
  • Naturally bounded [0, 1]

❌ Rejected Alternatives

  • L1 Normalization (scores sum to 1): Forces competition between categories. User genuinely interested in Travel AND Finance gets both scores suppressed. Bad for multi-interest users.
  • Min-Max Scaling: Sensitive to outliers. One power user with raw score of 500 compresses everyone else's scores to near-zero.
  • Z-Score: Assumes normal distribution of scores, but most category scores are heavily right-skewed (many zeros, few high values). Produces negative normalized scores, which are unintuitive.
  • Rank-Based: Similar to percentile but doesn't produce continuous scores — just rank ordering. Less useful for threshold-based segment assignment.

3.6.4 Audience Segment Assignment

The final step is converting normalized score arrays into actionable audience segments that advertisers can target. Each user is assigned to their top-K IAB audience segments based on their normalized scores.

Segment Assignment Logic
For user u with normalized score vector [ns_1, ns_2, ..., ns_700]: 1. FILTER: Remove categories where ns_c < 0.60 (Below 60th percentile = not a meaningful signal) 2. RANK: Sort remaining categories by ns_c descending 3. SELECT TOP-K: Take top K=15 categories (Cap prevents segment explosion; 15 covers most advertiser needs) 4. ASSIGN TIER: - "Strong Affinity": ns_c >= 0.90 (top 10% of population) - "Moderate Affinity": 0.75 <= ns_c < 0.90 - "Light Affinity": 0.60 <= ns_c < 0.75 5. OUTPUT: User segment membership list { "user_id": "device_abc123", "segments": [ {"category": "IAB20", "tier": "strong", "score": 0.92}, {"category": "IAB20-10", "tier": "strong", "score": 0.91}, {"category": "IAB13", "tier": "moderate", "score": 0.83}, {"category": "IAB12", "tier": "moderate", "score": 0.80}, {"category": "IAB3", "tier": "light", "score": 0.67} ], "total_segments": 5, "last_updated": "2026-03-30T14:22:05Z" }
Business Impact of Tiered Segments: Advertisers pay different CPM rates for different affinity tiers. A "Strong Affinity: Travel" user commands 4-5x CPM premium because they are highly likely to convert on a travel ad. A "Light Affinity: Travel" user gets a 1.5-2x premium. This tiered approach allows advertisers to control their spend based on confidence levels, and it increases our total addressable revenue by offering segments at multiple price points rather than a single binary "in/out" segment.

❄️ 3.8 Cold Start Problem & Sparse User Handling

Not every user arrives with a rich browsing history. Cold-start users — those with 1-2 observed domain visits — represent a significant fraction of our user base. In a given month, approximately 30% of the 500M tracked users have fewer than 5 observed visits, and 10% have only 1 observed visit. Assigning these users to audience segments based on minimal data risks creating noisy, unreliable segments.

3.8.1 The Cold Start Spectrum

User Type Observed Visits % of User Base Active IAB Categories Segment Confidence
Ice Cold 1 ~10% 3-8 (from single domain) Very Low - do not segment
Cold 2-4 ~20% 8-20 Low - contextual fallback preferred
Warm 5-20 ~35% 15-50 Medium - segment with lower confidence tier
Hot 20+ ~35% 30-100+ High - full segmentation

3.8.2 Cold Start Mitigation Strategies

1

Minimum Visit Threshold for Segment Eligibility

Users with fewer than 3 observed visits are excluded from audience segment assignment entirely. Their profiles still accumulate scores (so they "warm up" over time), but they are not sold to advertisers as targetable segments. This prevents advertisers from paying CPM premiums for unreliable signals.

2

Contextual Fallback for Cold Users

When a bid request arrives for a cold user, instead of using their sparse score array, we fall back to contextual targeting: we use the current domain's IAB classification as the targeting signal. "This user is currently on a Travel site" is a valid targeting signal even if we don't know their long-term interests. This contextual signal is included in the bid response as a separate signal type so advertisers can choose their trust level.

3

OpenRTB Metadata Enrichment

For cold users, we supplement their sparse profile with signals from the bid request itself: the bcat (blocked categories) and cat (content categories) fields from OpenRTB, device type, geo-location, and time-of-day patterns. A mobile user browsing at 10pm from a residential IP in a suburban area has a different interest profile baseline than a desktop user browsing at 2pm from a corporate IP.

4

Cohort-Based Priors

For cold users, we initialize their score array with a weak prior based on their demographic cohort. Users are grouped by (device_type, geo_region, time_zone) and each cohort has a pre-computed average IAB score distribution. For example, the cohort (mobile, US-Northeast, EST) might have a slight bias toward News and Finance. This prior is weighted at 0.1x the strength of an actual visit, so it provides a gentle nudge that is quickly overridden by real browsing data.

Failure Scenario: Over-Segmenting Cold Users. If we segment cold users aggressively (e.g., assigning "Travel Intender" to someone who visited one travel-adjacent domain), advertisers will see lower conversion rates from our segments. Over time, this erodes trust and reduces the CPM premium they are willing to pay. Our minimum-visit threshold is the primary defense. We track conversion rates segmented by user warmth tier and have an SLA that cold-user segments must maintain at least 60% of the conversion rate of hot-user segments, or the segment is gated.

Why Async Score Writes? The user score write-back to Redis is the most latency-variable operation (~1-3ms). By making it asynchronous (fire-and-forget), we remove it from the critical path. If the write fails (Redis timeout, network blip), the score update is lost for this single bid request — but with users generating 10-50+ bid requests per day, one missed update is negligible. We track async write failure rates and alert if they exceed 0.1% (indicating a Redis cluster health issue).

3.9.3 Failure Modes and Graceful Degradation

Every component in the inference path can fail. We design for graceful degradation so that a failure in one component doesn't crash the entire bid response pipeline.

Failure Scenario Impact Degradation Strategy Detection
Domain not in lookup table Can't classify user's current visit Fall back to OpenRTB cat field from bid request. If unavailable, skip score update for this visit. User retains their existing profile. Track "domain miss" rate. Alert if >8% of bid requests miss (indicates stale crawl list).
User has no profile (new user) No scores to normalize or segment Create new profile with this visit's domain scores. Use contextual targeting (current domain's categories) as the immediate signal. Apply cohort-based prior. Track new-user creation rate. Sudden spikes may indicate ID resolution issues.

Cached data is stale but usable for minutes. Redis health checks every 5 seconds. Auto-failover to replica within 30 seconds.
Redis user cluster down Can't fetch or update user profiles All users treated as cold-start. Contextual targeting only. Score updates queued in Kafka for replay when Redis recovers. Redis health checks. Score update Kafka queue depth monitoring.
Percentile histograms stale Normalization uses old distributions Histograms are rebuilt weekly but cached in-memory. If rebuild fails, use previous week's histograms (scores drift slowly, so week-old distributions are ~98% accurate). Histogram freshness check. Alert if histograms are >10 days old.
Latency exceeds 10ms budget Bid response too slow, SSP may time out Circuit breaker: if P99 exceeds 8ms for 60 seconds, skip score update and serve from cached segments only. This reduces latency to ~1ms. Real-time latency percentile monitoring. Auto-circuit-breaker activation.
Worst Case: Total Redis Failure. If both the domain cluster and user cluster are simultaneously unavailable (e.g., network partition), the bid server falls back to pure contextual targeting using only the OpenRTB metadata from the bid request. This means no personalized audience segments — just "the user is currently on a Travel site." While this eliminates the CPM premium from behavioral segments, it keeps the system operational and prevents bid response timeouts. In practice, this dual-failure scenario has occurred twice in 18 months, each time lasting <3 minutes before automated failover resolved it.

3.9.4 A/B Testing Hooks

The inference path includes built-in hooks for A/B testing different model configurations without redeploying:

Testable Parameter Mechanism Example Test
Decay rates Decay rate config stored in Redis feature flag. Test users get alternative lambda values. Test lambda_Travel = 0.015 (45-day half-life) vs production 0.023 (30-day). Measure conversion lift.
Classifier version Dual lookup tables in Redis (v3.2 and v3.3). User hash determines which table is used. Test MLP classifier (v3.3) vs LogReg (v3.2). Compare segment precision/recall.
Segment thresholds Threshold config per user bucket. Test stricter thresholds (fewer, higher-quality segments). Test threshold 0.75 vs 0.60. Does higher threshold increase CPM but reduce reach?
Normalization method Normalization function is swappable at runtime. Test group gets alternative normalization. Test Z-score normalization vs percentile. Which produces better advertiser ROI?

Key Insight: A/B testing in audience targeting requires special care because the feedback loop is long. A user assigned to a test segment today may not see (and click) a targeted ad for days or weeks. We use a 21-day test window minimum for all A/B tests, with a 14-day burn-in period to allow score arrays to converge under the new parameters before measuring outcomes. Statistical significance is computed using Bayesian methods rather than frequentist p-values, since conversion events are rare and asymmetric.

Memory Optimization: Sparse vs Dense User Arrays. Storing all 700 IAB categories per user would cost 700 * 4 bytes = 2.8 KB per user (as float32). At 500M users, that's 1.4 TB — exceeding reasonable Redis cluster sizes. Instead, we store only non-zero categories (average ~25 per user), reducing to ~500 bytes per user and ~250 GB total. The trade-off is slightly higher CPU cost for hash operations vs. array indexing, but the 5.6x memory savings make Redis Cluster feasible on 12 nodes instead of 70+.

🔗 3.11 Supplementing with OpenRTB Metadata

Our domain classification pipeline provides the primary intelligence for IAB categorization, but the OpenRTB bid request itself carries valuable metadata that can supplement or substitute for our classification in specific scenarios.

3.11.1 Available OpenRTB Category Fields

OpenRTB Field Description Availability Reliability
site.cat IAB content categories of the site ~65% of bid requests Medium — set by SSP, quality varies. Some SSPs use only Tier-1 categories.
site.pagecat IAB categories of the specific page ~25% of bid requests Higher than site.cat when available — page-level is more specific
site.sectioncat IAB categories of the site section ~15% of bid requests Medium — useful for large publishers with distinct sections (e.g., nytimes.com/sports)
bcat Blocked categories (advertiser restrictions) ~40% of bid requests High — tells us what the site is NOT, useful for filtering

3.11.2 Fusion Strategy

We combine our domain classification with OpenRTB metadata using a confidence-weighted fusion:

Score Fusion Logic
For each IAB category c in the bid request: IF domain is in our lookup table (95% of cases): final_score[c] = our_score[c] (trust our classification) IF openrtb_cat also available: Agreement check: - If our top-3 categories overlap with openrtb_cat: confidence = "high" - If no overlap: flag for review (possible misclassification or spoofing) IF domain NOT in lookup table (5% of cases): IF openrtb_cat available: final_score[c] = openrtb_score * 0.60 (discounted confidence) (Lower weight because SSP categories are less granular and less consistently applied than our classification) IF openrtb_cat NOT available: final_score = {} (no score update for this visit) Log event for domain discovery pipeline SPECIAL CASE: Multi-topic domains (reddit, medium, etc.): IF site.pagecat available (page-level category): Prefer pagecat over our domain-level classification final_score[c] = pagecat_score * 0.80 (Page-level is more precise than domain-level for aggregators)
Key Insight: OpenRTB metadata is a valuable supplement but not a replacement for our classification. SSP-provided categories are often coarse (Tier-1 only), inconsistently applied, and sometimes outright wrong. However, they serve three critical roles: (1) fallback when our lookup table misses a domain, (2) cross-validation signal for spoofing detection, and (3) page-level precision for multi-topic domains that our domain-level classification handles poorly.

🔒 3.12 Privacy & Compliance Considerations

Our model architecture must comply with GDPR, CCPA, and the IAB Transparency & Consent Framework (TCF). Privacy is not an afterthought — it is baked into every architectural decision.

Key Privacy Constraints on Architecture

Regulation Requirement Architectural Impact
GDPR (EU) Right to deletion; consent required for profiling User score arrays must be deletable within 72 hours. Redis TTL + DynamoDB/S3 deletion pipeline. Consent signal checked before score accumulation.
CCPA (California) Right to opt-out of sale of personal information Users with opt-out flag skip audience segment assignment entirely. Scores may still accumulate (for internal analytics) but are never exposed to advertisers.
TCF v2.2 Granular consent for different processing purposes Check TCF consent string in bid request. Purpose 1 (Store/access device info) and Purpose 4 (Personalization) must be consented before score accumulation.
Cookie Deprecation Chrome Privacy Sandbox; 3rd party cookie removal Architecture supports multiple ID types: cookies (legacy), device IDs (IDFA/GAID), probabilistic IDs (Unified ID 2.0, ID5). Score arrays are keyed by a canonical user ID that maps from any identifier type.
Failure Scenario: GDPR Deletion Request Not Propagated. If a user exercises their Right to Erasure and our system fails to delete their score array from Redis within 72 hours, we face potential fines up to 4% of global revenue. Mitigation: Deletion requests are published to a dedicated Kafka topic consumed by all storage tiers simultaneously. Redis deletion is immediate (within seconds). DynamoDB and S3 deletions are batched hourly. A reconciliation job runs daily, scanning for any user IDs in the deletion queue that still have residual data in any tier, and re-issues deletion commands.
